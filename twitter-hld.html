Clarifying Questions to Ask
Must Ask ▼
SCOPE
What features do we need to support?
Tweet posting, following, home timeline, search, trending, notifications, DMs, media uploads? Pin down the MVP scope.
SCALE
How many users are we designing for?
Get MAU/DAU, tweets per day, read:write ratio. "Twitter-like" could mean 10K users or 500M — confirms estimation direction.
PERFORMANCE
What are latency & availability expectations?
Is 200ms acceptable for timeline? Is it okay to show slightly stale data (eventual consistency)? Defines caching and consistency model.
CONSTRAINTS
Any special constraints?
Character limits, media file sizes, rate limiting, geolocation requirements, content moderation, celebrity / whale user handling.
FOCUS
Which area should I go deepest on?
Feed generation? Notification system? Search? This signals where to spend your deep-dive time.

Non-Functional Requirements

⚡
⚠️ NFRs drive the entire architecture. State them explicitly — they justify every design decision you make later (CDN, caching, sharding, fan-out strategy, etc.)
⚡
System Quality Attributes
NFR ▼
Attribute Requirement Target Justification
Availability System must stay up 99.99% (52m/yr downtime) Global social platform — outages are very public
Read Latency Timeline load <150ms p99 User retention drops sharply at >200ms
Write Latency Tweet posted <500ms p99 Acceptable for eventual consistency fan-out
Scalability Handle traffic spikes 10× surge capacity Breaking news events cause massive spikes
Consistency Feed freshness Eventual (~sec) Strong consistency not needed; stale feeds OK
Durability Zero data loss 11 nines (S3-class) Tweets are user-generated content; must not lose
Fault Tolerance Partial failures Graceful degradation Recommendation failure → show chronological feed
Security Auth, privacy OAuth2, E2E for DMs Account takeovers are high-profile incidents
Observability Metrics, tracing, logs Real-time dashboards Required for rapid incident response
📌 CAP Theorem Choice: Twitter chooses Availability + Partition Tolerance (AP) over Consistency. A user seeing a tweet appear 2 seconds late is acceptable; a user seeing "Service Unavailable" is not.

Counter Questions & Expected Answers

💬
🎯 Interviewers will probe your design. These are the most common challenges. Practice saying these answers out loud until they feel natural.
Q
How do you handle a famous user with 50 million followers posting a tweet?
▼
The "Celebrity Problem" / "Hot Key Problem"
  • Pure fan-out-on-write is impossible — 50M writes would take hours and spike the DB.
  • Use hybrid strategy: celebrity tweets are NOT fanned out. Instead, stored once in Cassandra.
  • At read time, timeline service identifies celebrity followees for this user, fetches their last N tweets, and merges with pre-computed feed from regular users.
  • Celebrity threshold: configurable (e.g., >10K followers = celebrity mode).
The interviewer wants to hear: "fan-out on write doesn't scale for celebrities, so I'd use pull for high-follower accounts and push for regular users — a hybrid approach."
Q
How do you ensure tweets appear in real-time? What about WebSockets vs polling?
▼
Real-time Feed Updates:
  • WebSockets (preferred): Persistent connection, server pushes new tweet_ids as they arrive. Client fetches tweet details. Efficient — no wasted polls.
  • Server-Sent Events (SSE): Simpler than WebSocket, HTTP-based, one-directional. Good for notification streams.
  • Long Polling: Fallback for clients that can't maintain WebSocket (corporate proxies).
  • Architecture: When fan-out worker writes to Redis timeline, it also publishes to a Redis pub/sub channel per user. WebSocket service subscribes per connected user.
Scale concern: 200M DAU can't all hold WebSocket connections. Use multiplexing — route to shared WebSocket servers partitioned by user_id range. Each server handles ~50K connections.
Q
What happens if your cache goes down? How do you prevent a cache stampede?
▼
Cache Failure Resilience:
  • Redis Cluster with replicas — automatic failover. No single point of failure.
  • If cache is cold (stampede risk): use probabilistic early expiration or mutex/distributed lock so only one request regenerates the cache while others wait.
  • Circuit breaker: if cache is fully down, fall back to database reads with rate limiting to prevent DB overload.
  • Gradual cache warm-up on startup: don't accept traffic until cache is pre-warmed from DB snapshots.
  • Cache-aside pattern (not write-through) gives more control over what gets cached.
Mention the "thundering herd" problem by name — it shows experience with distributed systems failure modes.
Q
How do you shard your database? What's your sharding key for tweets?
▼
Sharding Strategy:
  • Tweet sharding by tweet_id (Snowflake ID = timestamp + worker_id + sequence). This distributes writes evenly and allows time-range queries on a single shard.
  • User timeline sharding by user_id: All timeline entries for a user go to the same shard → no cross-shard joins for timeline fetch.
  • Hotspot concern: Sharding by user_id means a celebrity's data is on one shard. Mitigate with read replicas + caching.
  • Consistent hashing: Adds/removes shards without full reshuffling. Virtual nodes smooth load distribution.
Snowflake ID is Twitter's actual invention for this problem — mention it. It provides: globally unique IDs, monotonically increasing (sortable by time), generated without a DB round-trip.
Q
How do you handle the like/retweet count being slightly off?
▼
Eventual Consistency for Counters:
  • Exact counts are not critical — users don't notice "10,000 vs 10,003 likes."
  • Counts stored in Redis with INCR — atomic, fast. Periodically flushed to Cassandra (every 30s).
  • For display, read from Redis (fast, possibly slightly stale). For analytics, read from Cassandra.
  • If Redis node fails: recover counts from Cassandra + Kafka event replay (all like events are in Kafka).
This is a great example of trading strong consistency for performance. Say explicitly: "I choose AP over CP here — eventual consistency is acceptable for engagement counts."
Q
How would you scale search to handle 600K read QPS?
▼
Search Scaling:
  • Elasticsearch cluster with horizontal sharding — shard by time window (shard per day/week). Old shards are read-only and highly cacheable.
  • Search result caching: Popular queries (e.g., "World Cup") cached in Redis. Invalidate when new tweets with those terms arrive.
  • Read replicas: Multiple ES replicas serve search traffic. Write to primary, reads distributed.
  • Query optimization: Use filter clauses (not scored, cached by ES) for structured filters (date range, verified only). Only full-text on content field.
  • Separate recent vs. historical: Hot index (last 7 days) in SSD-backed nodes, cold index (older) in HDD-backed nodes.
Q
How do you handle rate limiting at scale?
▼
Distributed Rate Limiting:
  • Sliding window counter in Redis: Use ZADD to track request timestamps per user. O(log N) per check. Accurate but memory-intensive.
  • Token bucket: Simpler, allows bursting. Stored in Redis. Replenish tokens at fixed rate.
  • Fixed window: Simplest, slight boundary condition issue. Fine for most use cases.
  • Rate limit tiers: Unauthenticated (100 req/hr), Regular (1500 req/hr), Partner API (custom).
  • Edge enforcement: Rate limits checked at API Gateway — before hitting any downstream service.
  • Distributed: All gateway nodes share same Redis cluster → consistent limits across regions.
Mention X-RateLimit-Remaining, X-RateLimit-Reset headers in responses — shows you think about developer experience, not just backend.
Q
How do you design the "Who to Follow" recommendation system?
▼
Follow Recommendations:
  • Graph-based: "Friends of friends" (2nd-degree connections) — computed offline via Hadoop/Spark nightly, stored in Redis.
  • Interest-based: User's engagement history → infer interests → recommend users who tweet about those topics.
  • Collaborative filtering: Users with similar follow graphs get similar recommendations (matrix factorization).
  • Trending account signals: Accounts gaining followers fast in your network get boosted.
  • Serving: Pre-computed results cached per user, refreshed daily. Real-time blend with personalization layer.
Separate offline training (batch, Spark on HDFS) from online serving (pre-computed, Redis). Don't try to compute recommendations in real-time — it won't scale.

Key Concepts to Master

🧠
❄️
Snowflake ID
Twitter's distributed ID generation. 64-bit: 41 bits timestamp + 10 bits machine ID + 12 bits sequence. Globally unique, time-sortable, DB-free generation. ~4096 IDs/ms/machine.
ID Generation Distributed Time-sortable
💍
Consistent Hashing
Maps keys to nodes on a ring. Adding/removing nodes only remaps ~K/N keys (K=keys, N=nodes) instead of all keys. Virtual nodes balance load. Essential for elastic sharding.
Sharding Load Balancing Elasticity
🌐
CAP Theorem
In a distributed system, you can only guarantee 2 of 3: Consistency, Availability, Partition Tolerance. Networks always partition → choose CP (banks) or AP (social media). Twitter chooses AP.
Distributed Systems Consistency Availability
🌊
Back Pressure
When a consumer can't keep up with a producer, signal to slow down. Kafka consumers can lag behind, then catch up. Prevents OOM and cascading failures. Use bounded queues.
Resilience Kafka Flow Control
🎯
Count-Min Sketch
Probabilistic data structure for frequency estimation. Uses d×w bit matrix with d hash functions. O(1) insert/query, O(d×w) space. Perfect for counting hashtag occurrences for trending with bounded memory.
Probabilistic Trending Space-efficient
🔒
Distributed Locks (Redlock)
Redis-based distributed mutual exclusion. Acquire lock across N Redis nodes (majority). Used for cache stampede prevention, job deduplication. But: network partitions can cause issues — use with care.
Concurrency Redis Cache
📈
Read Replicas
Route all reads to replica DBs, writes to primary. Reduces primary load by N replicas. Slight replication lag (eventual consistency). Replicas can be geographically distributed for lower latency.
Scalability PostgreSQL MySQL
⚡
CQRS Pattern
Command Query Responsibility Segregation: separate write model (commands: post tweet) from read model (queries: get timeline). Allows independent scaling and optimization of each path. Twitter naturally follows this.
Architecture Pattern Scalability
🔄
Saga Pattern
Manage distributed transactions across microservices without 2-phase commit. Chain of local transactions, each publishing events. Compensating transactions handle rollbacks. Used for tweet deletion propagation.
Microservices Transactions Consistency
🛡
Circuit Breaker
Prevent cascading failures: CLOSED (normal) → OPEN (failures exceed threshold, fast-fail) → HALF-OPEN (test if service recovered). Used around downstream service calls. Netflix Hystrix popularized this.
Resilience Fault Tolerance Pattern
🗜
Write-Ahead Log (WAL)
All changes written to append-only log before applying to storage. Enables crash recovery, replication, CDC (change data capture). Kafka itself is essentially a distributed WAL. Used by PostgreSQL and Cassandra.
Durability Replication Recovery
🎪
Service Mesh (Istio)
Sidecar proxies (Envoy) handle service-to-service communication: mTLS, load balancing, circuit breaking, observability. Services communicate without knowing about infrastructure concerns. Reduces boilerplate in every service.
Kubernetes Microservices Observability
