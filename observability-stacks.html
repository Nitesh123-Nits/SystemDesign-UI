ELK STACK
Elasticsearch · Logstash · Kibana — The original centralized logging platform
LOGGING
ElasticsearchLogstash KibanaFilebeatMetricbeat
Full-text search-grade log aggregation. Logstash ingests, parses, and transforms; Elasticsearch indexes in near-real-time with inverted index; Kibana visualizes with KQL queries, dashboards, and APM.
▸ DATA FLOW
01App/Filebeat ships logs → Logstash :5044 (Beats input)
02Logstash parses (Grok), filters, enriches (GeoIP, DNS)
03Output to Elasticsearch index (logs-YYYY.MM.DD)
04Kibana queries via REST API, renders dashboards
05Watcher triggers alerts on query conditions
ES :9200 RESTES :9300 cluster Logstash :5044Kibana :5601

EFK STACK
Elasticsearch · Fluentd · Kibana — Kubernetes-native logging
K8S LOGGING
ElasticsearchFluentd Fluent BitKibana
Kubernetes-first logging. Fluent Bit (lightweight) runs as DaemonSet collecting container logs from /var/log/containers; Fluentd aggregates and routes; Elasticsearch stores. Lower memory footprint than Logstash.
▸ DATA FLOW
01Fluent Bit DaemonSet reads /var/log/containers/*.log
02Tail plugin → Kubernetes metadata filter enriches
03Forward to Fluentd aggregator (tag-based routing)
04Fluentd parses, buffers → Elasticsearch output
05Kibana visualizes; Index Lifecycle Management (ILM)
Fluentd :24224Fluent Bit :2020 ES :9200Kibana :5601

GRAFANA LGTM
Loki · Grafana · Tempo · Mimir — Full observability platform
ALL-IN-ONE
GrafanaLoki TempoMimirAlloy
The LGTM stack unifies logs (Loki), metrics (Grafana + Mimir), and traces (Tempo) with correlations. Grafana Alloy replaces Grafana Agent as the unified telemetry collector. Jump from log line → trace → metric.
▸ DATA FLOW
01Grafana Alloy/Agent collects logs, metrics, traces
02Logs → Loki (label index, chunk store S3)
03Metrics → Mimir / Prometheus remote_write
04Traces → Tempo (object store: S3/GCS)
05Grafana correlates: logs ↔ traces (TraceID in logs)
Grafana :3000Loki :3100 Tempo :3200Mimir :9009

OPENTELEMETRY
Vendor-neutral observability framework — CNCF standard
FRAMEWORK
OTel SDKOTel Collector OTLP protocolAuto-InstrumentSemantic Conventions
CNCF standard for generating and collecting telemetry. Replaces Jaeger/Zipkin clients. SDK instruments apps → OTel Collector (Receiver → Processor → Exporter) → any backend. Supports all three signals: traces, metrics, logs.
▸ DATA FLOW
01Auto-instrumentation (Java Agent, Node SDK) attaches
02SDK creates spans, records metrics, emits log records
03OTLP export → OTel Collector :4317 (gRPC) / :4318 (HTTP)
04Collector: batch, memory_limiter, attributes processors
05Export to Tempo (traces), Prometheus (metrics), Loki (logs)
OTLP gRPC :4317OTLP HTTP :4318 Collector :55679 (zpages)

GRAFANA TEMPO
Cost-efficient distributed tracing — no indexing, pure object storage
TRACING
DistributorIngester QuerierCompactorTraceQL
Unlike Jaeger/Zipkin which require Cassandra/ES, Tempo stores traces directly in object storage (S3/GCS) with no indexing cost. TraceQL for structured trace queries. Integrates with Grafana for log-to-trace correlation via TraceID.
▸ DATA FLOW
01OTel Collector / Grafana Alloy sends traces (OTLP/Jaeger)
02Distributor receives, hashes by TraceID
03Ingester buffers in WAL → flushes to object store
04Compactor merges blocks, applies retention
05Querier fetches from object store on TraceID lookup
06Grafana renders: timeline, spans, service map
HTTP :3200gRPC :9095 OTLP :4317Jaeger :14268

⚡ SIGNAL CORRELATION — The Power of Unified Observability
LOG → TRACE
Inject TraceID into log lines (MDC in Java). In Grafana, configure Derived Fields on Loki datasource → clicking TraceID jumps directly to Tempo trace.
METRIC → TRACE (EXEMPLARS)
Prometheus Exemplars embed a TraceID sample into metric data points. When you see a latency spike in Grafana, click the exemplar dot → lands on the exact Tempo trace that caused it.
TRACE → LOG → METRIC
From Tempo span view, click "Logs for this span" → filtered Loki view. Click service → pre-filtered Grafana dashboard for that service's RED metrics (Rate, Errors, Duration).
