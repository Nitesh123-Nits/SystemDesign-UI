📨
Producer
Publishes records to a named topic. Uses a partitioner (default: round-robin or key hash) to decide which partition gets the record. Batches records for throughput via linger.ms and batch.size. Can use transactions for exactly-once semantics.
acks=all idempotent linger.ms batch.size compression.type
📋
Topic
Logical feed name. A topic is split into partitions — ordered, immutable sequences of records. Each record gets a monotonically increasing offset within its partition. Topics have a configurable retention policy (time or size-based).
num.partitions replication.factor retention.ms cleanup.policy
🗂️
Partition + Offset
The atomic unit of parallelism. Each partition is an append-only log stored on disk. Offset = position of a record within a partition. Consumers commit offsets to __consumer_offsets topic, enabling independent replay by different groups at any time.
append-only sequential I/O replayable
💾
Broker
A Kafka server process. Brokers form a cluster; each partition has one leader broker and N-1 follower replicas (ISR — In-Sync Replicas). Leader handles all reads/writes; followers replicate. Cluster metadata is managed by KRaft (formerly ZooKeeper).
leader/follower ISR KRaft log.dirs min.insync.replicas
👥
Consumer Group
Each consumer group reads the topic independently. Within a group, each partition is owned by exactly one consumer — enabling horizontal scaling. Multiple groups can consume the same topic simultaneously at different offsets (fan-out reads). Rebalancing is triggered when consumers join/leave.
group.id auto.offset.reset max.poll.records session.timeout.ms
🔗
Kafka Connect + Schema Registry
Connect is a framework for streaming data between Kafka and external systems (DBs, S3, Elasticsearch) using source and sink connectors — no custom code needed. Schema Registry enforces Avro/JSON/Protobuf schema contracts so producers and consumers stay compatible.
CDC Debezium Avro schema evolution
🌊
Kafka Streams / ksqlDB
Streams is an embedded Java library for stateful stream processing (joins, aggregations, windowing) that runs inside your app — no separate cluster needed. ksqlDB lets you write SQL queries directly on topics, materializing results as new topics or queryable tables.
KTable KStream windowing ksqlDB

📤
Publisher
Sends messages to an Exchange (never directly to a queue). Each message includes a routing key — a dot-separated string (e.g., order.created.eu). Publishers can declare messages as persistent (survives broker restart) or transient. Confirms delivery via publisher confirms.
routing key persistent publisher confirms mandatory flag
🔀
Exchange
The routing brain of RabbitMQ. Four types: Direct (exact key match → one queue), Fanout (ignores key → all bound queues), Topic (* = one word, # = zero or more words), Headers (matches on message header attributes, not the key). Each Exchange has a list of bindings.
direct fanout topic headers bindings
📬
Queue
Ordered buffer holding messages until consumed. Key properties: durable (survives restart), exclusive (tied to one connection), auto-delete (removed when last consumer leaves), TTL (message/queue expiry), max-length (overflow policy: drop-head or reject-publish), priority levels (0–255).
durable TTL priority max-length quorum queue
🔗
Binding
A rule linking an Exchange to a Queue. For Topic exchanges, the binding key uses wildcards: order.* matches order.created but not order.created.eu. A single queue can have multiple bindings from multiple exchanges — enabling complex routing topologies.
binding key wildcard * wildcard #
👷
Consumer + ACK/NACK
Consumers register on a queue and receive messages via push (broker delivers). Must explicitly ACK each message after processing — unacknowledged messages are redelivered on consumer crash. NACK with requeue=false routes to the Dead Letter Exchange (DLX). prefetch_count limits unacked messages per consumer.
ack nack prefetch_count DLX requeue
☠️
Dead Letter Exchange (DLX)
Messages are dead-lettered when: NACK'd with requeue=false, TTL expires, or queue is full. Dead-lettered messages are forwarded to the DLX with their original routing key (or a new one via x-dead-letter-routing-key). Used for retry queues, error audits, and poison message handling.
x-dead-letter-exchange retry queue poison message
🏛️
Virtual Hosts (vhosts) + Quorum Queues
Vhosts are isolated namespaces within one broker — separate exchanges, queues, and permissions. Ideal for multi-tenant environments. Quorum Queues (Raft-based, replicated) replace classic mirrored queues and provide strong durability and consistency guarantees across a cluster.
vhost Raft quorum queue ha-policy

KEY SETTINGS EXPLAINED
min.insync.replicas
Durability guaranteeMinimum ISR brokers that must ACK a write before the producer gets success. Set to 2 with RF=3 for strong durability without sacrificing too much throughput.
log.retention.hours
Message lifetimeHow long Kafka keeps records regardless of consumption. Set to -1 (unlimited) for event sourcing / audit log patterns. Use log.cleanup.policy=compact for changelog topics.
process.roles
KRaft modeIn KRaft (Kafka 3.3+), brokers can act as both broker and controller, eliminating the ZooKeeper dependency entirely. Always use KRaft for new deployments.
num.partitions
Parallelism ceilingDefault partition count for auto-created topics. Rule of thumb: partitions ≥ (target throughput MB/s) ÷ (throughput per partition). Over-partitioning wastes resources; under-partitioning limits scale.
log.cleanup.policy
Retention strategydelete = remove old segments after time/size limit. compact = keep only the latest record per key (ideal for state/changelog topics like a DB table mirror).

KEY SETTINGS EXPLAINED
acks=all
Strongest durabilityWaits for ALL in-sync replicas to write the record. Required when enable.idempotence=true. acks=1 is faster but risks data loss on leader failure. acks=0 is fire-and-forget.
enable.idempotence
Exactly-once per partitionProducer gets a PID + sequence number; broker deduplicates retries. Add transactional.id to extend this across partitions and topics (full EOS).
linger.ms + batch.size
Throughput vs latency tradeoffHigher linger.ms = more records per batch = better compression + fewer requests. For low-latency systems use linger.ms=0. For high-throughput pipelines use 5–20ms.
compression.type
Bandwidth savingslz4 is the best balance of CPU vs ratio. zstd gives the highest ratio. snappy has the lowest CPU overhead. Compression is applied per batch, so larger batches compress better.
transactional.id
Atomic multi-partition writesEnables read-process-write loops with exactly-once semantics across topics. Each producer instance must have a unique ID. Broker uses this to fence zombie producers.

KEY SETTINGS EXPLAINED
enable.auto.commit=false
Manual offset controlAlways disable in production. Commit offsets only after successful processing to avoid data loss (crash before commit) or duplication (crash after process but before commit).
auto.offset.reset
New group starting pointearliest = replay from beginning (use for new consumers). latest = start from now (use for real-time-only pipelines). none = throw exception if no offset found.
max.poll.interval.ms
Rebalance protectionIf poll() isn't called within this window, the consumer is kicked from the group and rebalance triggers. Must be longer than your processing time per batch. Common cause of unexpected rebalances.
isolation.level
Transactional readsread_committed ensures consumers only see records from committed transactions. Required on the consumer side to complete an end-to-end exactly-once pipeline.
CooperativeStickyAssignor
Incremental rebalancingOnly moves partitions that need to move during rebalance — existing assignments stay active. Eliminates the "stop-the-world" rebalance of the default Eager assignor. Use this in production.

KEY SETTINGS EXPLAINED
vm_memory_high_watermark
Flow control thresholdWhen memory usage exceeds this fraction of total RAM, RabbitMQ blocks publishers (backpressure). Set 0.4–0.7 depending on your server. Too low = unnecessary blocking; too high = OOM kills.
disk_free_limit
Disk alarmIf free disk space drops below this, all publishing is blocked to protect message durability. Should be at least as large as your largest expected message backlog.
quorum_queue.delivery_limit
Poison message protectionAfter this many redeliveries, the message is dead-lettered instead of retried indefinitely. Prevents a single bad message from blocking a queue worker loop forever.
heartbeat
Connection healthBroker and client exchange heartbeats at this interval. If 2 consecutive heartbeats are missed, the connection is considered dead. Critical for detecting silent TCP disconnects.
channel_max
Multiplexing limitAMQP multiplexes multiple channels on one TCP connection. Each channel is an independent flow. Keep this reasonable — too many channels per connection increases broker memory usage.
topology-declaration.js (amqplib)
RABBITMQ 
// ── Connection ────────────────────────────
const conn = await amqp.connect({
  protocol: 'amqps',
  hostname: 'rabbitmq.prod.internal',
  port: 5671,
  vhost: '/production',
  username: 'order-service',
  password: process.env.RABBIT_PASS,
  heartbeat: 60,
});
const ch = await conn.createConfirmChannel();

// ── Exchange Declaration ───────────────────
await ch.assertExchange('orders', 'topic', {
  durable: true,       // survives broker restart
  autoDelete: false,
  alternateExchange: 'orders.unrouted', // catch unrouted msgs
});

// ── Queue Declaration ─────────────────────
await ch.assertQueue('order.email.notifications', {
  durable: true,
  arguments: {
    'x-queue-type':            'quorum',   // Raft-based HA
    'x-dead-letter-exchange':  'orders.dlx',
    'x-dead-letter-routing-key': 'failed.email',
    'x-message-ttl':           86400000,  // 24h TTL
    'x-max-length':            100000,    // max 100K msgs
    'x-overflow':              'reject-publish-dlx',
  },
});

// ── Binding ───────────────────────────────
await ch.bindQueue(
  'order.email.notifications',
  'orders',
  'order.#'   // matches order.created, order.shipped.eu, etc
);

// ── Consumer with prefetch ────────────────
await ch.prefetch(20); // max 20 unacked msgs per consumer
ch.consume('order.email.notifications', async (msg) => {
  try {
    await sendEmail(msg.content);
    ch.ack(msg);                  // success → delete
  } catch (err) {
    const requeue = !msg.fields.redelivered;
    ch.nack(msg, false, requeue); // retry once, then DLX
  }
}, { noAck: false });          // always use manual ACK
KEY SETTINGS EXPLAINED
x-queue-type: quorum
Replicated durabilityQuorum queues use Raft consensus across cluster nodes. Data is only confirmed written when a majority of nodes have it. Use instead of classic mirrored queues for all new deployments.
x-dead-letter-exchange
Failure handlingWhen a message is NACK'd, expired, or the queue overflows, it's forwarded here. Essential for retry queues, poison message inspection, and audit trails of failed processing.
prefetch(20)
Fair dispatchLimits unacknowledged messages per consumer. Without this, RabbitMQ sends all queued messages to the first available consumer, causing imbalance. Start at 5–20 and tune based on processing time.
order.#
Topic wildcard binding# matches zero or more dot-separated words. order.# matches order.created, order.shipped.eu, and order itself. Use * for exactly one word segment.
alternateExchange
Catch unroutable messagesIf a published message matches no binding, it goes here instead of being silently dropped. Set up a logging queue bound to this exchange to detect misconfigured routing keys.
