01 — Foundations

Core Concepts

The building blocks of every distributed system — understanding these deeply is the difference between systems that scale and systems that fail.

△
Foundation
CAP Theorem
Any distributed data store can provide only two of three guarantees: Consistency, Availability, and Partition tolerance. Brewer's theorem shapes every architecture decision.
Brewer 2000 Proven 2002 Trade-offs

Data Properties
ACID vs BASE
ACID provides strong guarantees at the cost of availability. BASE trades consistency for availability — Basically Available, Soft state, Eventually consistent.
Transactions Consistency NoSQL

FLP Impossibility
In an asynchronous distributed system, no deterministic consensus protocol can tolerate even one process crash failure. Fischer, Lynch, and Paterson, 1985.
1985 Turing Award Consensus

Vector Clocks
Track causality in distributed systems where there's no global clock. Each node maintains a vector of logical timestamps to determine event ordering and detect conflicts.
Lamport 1978 Causality Ordering

Information Dissemination
Gossip Protocol
Epidemic-style information spreading where each node periodically exchanges state with random neighbors. O(log N) dissemination with eventual consistency and fault tolerance.
Epidemic Anti-entropy Failure Detection

CRDT
Conflict-free Replicated Data Types automatically resolve conflicts using mathematical properties. Counters, sets, maps that merge without coordination — used by Redis, Riak, Apple's Notes.

Two Generals
Two armies must coordinate an attack, but communication may fail. Proves it's impossible to achieve guaranteed consensus over an unreliable channel — the foundation of TCP's 3-way handshake.

Raft
Designed to be understandable. Raft decomposes consensus into leader election, log replication, and safety. A leader handles all client requests and replicates entries to followers. Majority quorum required for commitment.
✓ Strong leader — single point of coordination
✓ Handles f failures with 2f+1 nodes
✓ Log matching: if two logs have same index+term, all prev entries identical
✗ Leader is bottleneck for all writes
✗ Leader election can cause brief unavailability

Paxos
The original consensus algorithm. Two-phase protocol: Prepare/Promise establishes a ballot number, Accept/Accepted commits a value. Multi-Paxos optimizes by electing a distinguished proposer for multiple rounds.

PBFT
Practical Byzantine Fault Tolerance handles nodes that can behave arbitrarily — sending conflicting messages, lying, or colluding. Required for blockchain and high-security systems. O(n²) message complexity.
✓ Handles Byzantine (malicious) failures
✓ Tolerates f failures with 3f+1 nodes
✓ Used in Hyperledger, early blockchain systems
✗ O(n²) communication — doesn't scale beyond ~100 nodes
✗ View changes are complex

Two-Phase Commit
Coordinator asks all participants to vote on committing a transaction. If all vote yes, commit; if any votes no, abort. Simple but blocking — if coordinator crashes after prepare, participants are stuck.
✓ Atomicity: all commit or none do
✓ Simple to understand and implement
✗ Blocking: coordinator SPOF
✗ Long-held locks reduce throughput
✗ Cannot progress during coordinator failure
Three-Phase Commit
Adds a pre-commit phase to 2PC to eliminate the blocking problem. CanCommit → PreCommit → DoCommit. If coordinator fails after PreCommit, participants can safely commit. However, not safe under network partitions.
✓ Non-blocking (unlike 2PC)
✓ Coordinator failure recoverable
✗ Unsafe under network partitions
✗ 3x the message rounds vs 2PC
✗ Rarely used in practice
ZAB — ZooKeeper Atomic Broadcast
ZooKeeper's consensus protocol. Leader broadcasts proposals to a quorum of followers. Guarantees total order delivery with epoch-based leader election. Optimized for high-throughput read-heavy workloads typical in coordination services.
✓ Primary-order: all writes go through leader
✓ Recovery: new leader recovers all committed proposals
✓ Used in production at Yahoo, Alibaba, Netflix
✗ Complex recovery protocol
✗ Write throughput limited by leader

06 — Resilience

Failure Patterns

Systems fail in predictable ways. Understanding failure modes — and their solutions — is what separates systems that recover gracefully from those that cascade into outages.

⚡
Split Brain
Network partition causes multiple nodes to become leaders. Writes accepted by both create irreconcilable divergence.
→ Fix: Quorum, STONITH, fencing tokens
🌊
Cascading Failures
One failing service overloads its dependencies, causing progressive system-wide collapse.
→ Fix: Circuit breakers, bulkheads, backpressure
🐘
Thundering Herd
Many processes wake simultaneously after a blocking event and overwhelm a downstream resource.
→ Fix: Jitter, mutex locks, request coalescing
🚧
Head-of-Line Blocking
Slow requests block faster ones waiting behind them in a queue or connection pool.
→ Fix: HTTP/2 multiplexing, QUIC, priority queues
⏱
Clock Skew
Machine clocks drift apart, causing ordering bugs, expired tokens, or causality violations.
→ Fix: Logical clocks, HLC, TrueTime
👹
Byzantine Failure
Nodes send arbitrary, conflicting, or malicious messages rather than simply crashing.
→ Fix: PBFT, cryptographic signing, BFT protocols
🔄
Metastable Failures
Stable bad state maintained by feedback loops even after original trigger is removed.
→ Fix: Backoff, retry budgets, load shedding
🌫
Gray Failures
Partial degradations that look healthy from some vantage points but cause user-facing errors.
→ Fix: p99 health checks, synthetic monitoring

08 — Data Architecture

Distributed Storage

Where data lives, how it's organized, and how it scales. From LSM trees powering RocksDB to the wide-column stores of Cassandra and BigTable.

◧
Write-Optimized
LSM Tree
Log-Structured Merge Tree. All writes go to in-memory memtable, flushed to immutable SSTables. Compaction merges SSTables. Powers LevelDB, RocksDB, Cassandra, HBase.
Write-heavyCompactionSSTables
🌲
Read-Optimized
B+ Tree
Balanced tree with sorted keys in leaf nodes, all leaves linked for range scans. O(log n) reads and writes. Used by PostgreSQL, MySQL InnoDB, SQLite, most RDBMS.
Read-heavyRange ScansRDBMS
◉
Probabilistic
Bloom Filter
Space-efficient probabilistic structure to test set membership. False positives possible, false negatives impossible. Used to avoid expensive disk lookups in LSM stores.
ProbabilisticO(1) lookupNo false negatives
📝
Durability
Write-Ahead Log
Append-only log written before applying changes to data. Enables crash recovery — replay WAL to restore state. Used in PostgreSQL, MySQL, etcd, ZooKeeper.
DurabilityRecoverySequential I/O
🔗
Distributed Transactions
Saga Pattern
Long-lived transactions as a sequence of local transactions. Each step publishes events. Compensating transactions undo completed steps on failure. No distributed locks needed.
MicroservicesCompensationChoreography
△
Data Integrity
Merkle Tree
Hash tree where leaves are data hashes and internal nodes are hashes of children. Efficient verification of distributed data consistency. Foundational to Git, Bitcoin, AWS DynamoDB anti-entropy.
IntegrityAnti-entropyGit, Bitcoin
