The Universal HLD Framework

A structured approach to answer any system design question in 45–60 minutes

01
Clarify Requirements
Spend 5–8 min asking clarifying questions. Never jump into design without scope.
  • Who are the users? Scale?
  • Read-heavy or write-heavy?
  • Functional requirements (must-haves)
  • Non-functional: latency, availability, consistency
  • What to NOT design (out of scope)
02
Capacity Estimation
Back-of-envelope math shows you understand real-world scale and constraints.
  • DAU / MAU → QPS calculation
  • Storage: per-entity × entities/day × retention
  • Bandwidth: read QPS × avg response size
  • Memory: hot data percentage of total
  • Peak vs average load (3–10× multiplier)
03
High-Level Design
Draw the birds-eye architecture: clients, API layer, services, databases.
  • Client → CDN/LB → API Gateway
  • Core services (what they do, not how)
  • Data stores: which type and why
  • Async flows: queues, workers
  • External integrations (push, email, etc.)
04
Deep Dive
Let the interviewer guide focus. Go deep on the hardest/most interesting part.
  • Data model & schema design
  • API design (REST/gRPC endpoints)
  • Critical algorithms (ranking, feed gen)
  • Bottlenecks & optimizations
  • Failure modes & recovery
05
Scale & Reliability
Show you can take the design from MVP to planet-scale with reliability guarantees.
  • Horizontal scaling strategy
  • Caching layers (L1, L2, CDN)
  • Database sharding / replication
  • Rate limiting & throttling
  • 99.99% uptime: circuit breakers, retries
06
Trade-offs & Summary
Close with self-awareness: what you chose and what you sacrificed.
  • CAP theorem position (CP vs AP)
  • SQL vs NoSQL justification
  • Sync vs async trade-offs
  • What you'd improve with more time
  • Monitoring & observability gaps

Numbers Every Engineer Must Know

Latency, throughput, and scale references for estimation

Latency Reference
L1 cache: 1ns · RAM: 100ns · SSD: 0.1ms · HDD: 10ms · Network round-trip: 1–500ms
Storage Units
1 char = 1 byte · Tweet = 280B · Photo = 300KB · Video 1min = 50MB · 1M users/day = ~100GB/day content
QPS Conversions
1M DAU × 10 req/day = ~115 QPS · 100M DAU = 11,500 QPS · 1B DAU = 115K QPS
Availability SLAs
99% = 3.6 days/yr · 99.9% = 8.7 hrs/yr · 99.99% = 52 min/yr · 99.999% = 5 min/yr
DB Throughput
Single SQL DB: ~1K QPS writes, ~5K reads · With replica: 20K reads · Sharded: linear scale
Cache Ratios
20% of data = 80% of traffic (Pareto). Aim for >90% cache hit rate. Redis: ~100K ops/sec

Design a Distributed Key-Value Store
HARD
AmazonGoogleUber
▶ EXPAND DESIGN
Architecture Pillars
Consistent Hashing
Virtual nodes on a ring. Key → hash → clockwise nearest node. Minimize reshuffling on node add/remove.
Replication
N=3 replicas clockwise on ring. Quorum: W+R > N for strong consistency (e.g., W=2, R=2, N=3)
Conflict Resolution
Vector clocks to track causality. Last-write-wins (LWW) for simplicity. Merkle trees for anti-entropy sync.
Failure Handling
Gossip protocol for membership. Hinted handoff for temporary failures. Read repair for stale replicas.
Consistent Hashing Quorum (W+R>N) Vector Clocks Gossip Protocol Merkle Trees Hinted Handoff

Hardest Problem: Double Booking Prevention
Optimistic Locking
Check availability → reserve with version check → commit. Retry on conflict. Good for low contention.
Pessimistic Locking
SELECT FOR UPDATE on date ranges. Guaranteed no double booking. Slower, may cause deadlocks.
Distributed Lock
Redis Redlock on property+date. TTL=30s. Atomic via Lua script. Best for microservices architecture.
Idempotency
Idempotency key per booking request. Prevent duplicate charges on retry. Store in DB with TTL.

Design a Distributed Message Queue (Kafka)
HARD
LinkedInUberConfluent
▶ EXPAND DESIGN
Log-based Storage
Append-only log on disk. Segments are immutable files. Offset per consumer group tracks position. Sequential I/O = very fast.
Partitions
Topic → N partitions across brokers. Partition key routes same user to same partition (ordering). More partitions = more parallelism.
Replication
Each partition has 1 leader + N-1 followers. Leader handles all reads/writes. Followers replicate via fetch protocol. ISR set.
Consumer Groups
Each group tracks its own offset. Multiple groups = multiple independent consumers. Group rebalance on member change.
Log Compaction ISR (In-Sync Replicas) ZooKeeper / KRaft Consumer Group
Design a CDN
HARD
CloudflareAkamaiAmazon CF
▶ EXPAND DESIGN
Anycast Routing
BGP anycast: same IP announced from all PoPs. Client DNS resolves to nearest PoP automatically. Reduces RTT dramatically.
Cache Hierarchy
L1 (edge PoP) → L2 (regional) → Origin. Cache miss propagates up. L2 acts as shield preventing origin overload.
Cache Invalidation
TTL-based expiry. Purge API for instant invalidation. Surrogate keys (cache tags) for grouped invalidation.
TLS Termination
TLS at edge PoP. Reduces latency: TLS handshake is local. Origin connection via persistent keep-alive.
Anycast BGP Cache-Control Headers TLS Offloading Origin Shield

Core Concepts Reference

Every critical concept you must know for HLD interviews — organized by category

Data Storage
🗄️
SQL vs NoSQL
SQL: ACID, joins, structured. NoSQL: horizontal scale, flexible schema. Choose SQL for transactions, NoSQL for scale/flexibility.
PostgreSQLDynamoDBCassandra
🔀
Database Sharding
Horizontal partitioning by shard key (user_id, geography). Consistent hashing minimizes reshuffling. Watch for hot shards.
Range-basedHash-basedDirectory
📡
Replication
Master-slave: 1 write node, N read replicas. Replication lag = eventual consistency. Multi-master = conflict resolution needed.
AsyncSyncSemi-sync
📊
ACID vs BASE
ACID: SQL guarantees. BASE: Basically Available, Soft state, Eventually consistent. NoSQL trades consistency for availability.
AtomicityConsistencyIsolation
🔍
Indexing
B-tree for range queries, Hash for equality. Composite index column order matters. Too many indexes slow writes.
B-treeLSM-treeInverted
⏱️
Time-Series DB
Optimized for append-only, time-ordered data. Column compression (same timestamps). Auto-downsampling. InfluxDB, TimescaleDB.
InfluxDBPrometheusDruid
Caching
⚡
Cache-Aside (Lazy Loading)
App checks cache → miss → fetch DB → populate cache. Cache only hot data. Risk: cache stampede on cold start.
RedisMemcached
✍️
Write-Through Cache
Write to cache AND DB simultaneously. Always fresh data in cache. Higher write latency. Good for read-heavy.
ConsistencyLatency++
📝
Write-Behind (Write-Back)
Write to cache only, async flush to DB. Fast writes, risk of data loss if cache crashes before flush.
PerformanceRisk
💥
Cache Stampede
Many requests hit DB simultaneously on cache expiry. Solutions: mutex lock, probabilistic early expiry, background refresh.
Thundering herdMutex
🌊
Eviction Policies
LRU (least recently used) — standard. LFU (frequency) — for viral content. TTL for freshness. FIFO simplest.
LRULFUTTL
🔒
Distributed Lock (Redlock)
Redis multi-node lock algorithm. Acquire on N/2+1 nodes, TTL safety net. Release via Lua script atomically.
MutexTTLAtomic
Scalability & Reliability
🔺
CAP Theorem
Consistency, Availability, Partition Tolerance — pick 2. During partition: choose CP (banking) or AP (social media).
CP: HBaseAP: Cassandra
🌀
Consistent Hashing
Virtual nodes on a ring. Adding/removing nodes redistributes only K/N keys instead of all. Foundation of DynamoDB, Cassandra.
Virtual NodesRing
⚖️
Load Balancing
Algorithms: Round-robin, Least connections, IP hash (sticky sessions), Weighted. L4 (TCP) vs L7 (HTTP content-aware).
NginxHAProxyAWS ALB
🔌
Circuit Breaker
Prevent cascade failures. States: Closed → Open → Half-open. After N failures, open circuit and fail-fast. Hystrix pattern.
Resilience4jHystrix
📦
Message Queue
Decouple producers/consumers. Async processing. Buffer traffic spikes. At-least-once vs exactly-once delivery semantics.
KafkaRabbitMQSQS
🎯
Service Mesh
Sidecar proxy (Envoy) per service. Handles: mTLS, retries, circuit breaking, observability. Control plane = Istio.
IstioEnvoymTLS
Design Patterns
📖
CQRS
Command Query Responsibility Segregation. Separate read/write models. Write model optimized for writes, read model for queries.
Event-DrivenRead Model
📜
Event Sourcing
Store state as sequence of events, not current state. Rebuild state by replaying events. Perfect audit trail. Kafka as event log.
Immutable LogReplay
🔗
Saga Pattern
Distributed transaction via choreography or orchestration. Each step emits event. Compensating transactions on failure.
ChoreographyOrchestration
🔑
Idempotency
Same request twice = same result. Idempotency key in header. Store processed request IDs with TTL. Critical for payments.
UUID keyDeduplication
🏭
Bulkhead Pattern
Isolate critical resources (thread pools, connections). Failure in one bulkhead doesn't drain shared resource pool.
IsolationResilience
🔄
Two-Phase Commit
Coordinator asks all participants to prepare, then commit. Guarantees atomicity. Slow (blocking protocol). Rarely used today.
2PCXA Protocol
