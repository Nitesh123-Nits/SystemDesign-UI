KafkaProducerConfig.java

@Configuration
@EnableKafka
public class KafkaProducerConfig {

  @Value("${spring.kafka.bootstrap-servers}")
  private String bootstrapServers;

  // ── Producer factory (non-transactional) ──────
  @Bean
  public ProducerFactory<String, Object> producerFactory() {
    Map<String, Object> props = new HashMap<>();
    props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, bootstrapServers);
    props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG,   StringSerializer.class);
    props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, JsonSerializer.class);
    props.put(ProducerConfig.ACKS_CONFIG,                   "all");
    props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG,      true);
    props.put(ProducerConfig.RETRIES_CONFIG,                Integer.MAX_VALUE);
    props.put(ProducerConfig.LINGER_MS_CONFIG,               10);
    props.put(ProducerConfig.BATCH_SIZE_CONFIG,             65536);
    props.put(ProducerConfig.COMPRESSION_TYPE_CONFIG,       "lz4");
    props.put(JsonSerializer.ADD_TYPE_INFO_HEADERS,         false);
    return new DefaultKafkaProducerFactory<>(props);
  }

  // ── KafkaTemplate (use this in services) ──────
  @Bean
  public KafkaTemplate<String, Object> kafkaTemplate() {
    return new KafkaTemplate<>(producerFactory());
  }

  // ── Topic auto-creation ───────────────────────
  @Bean
  public NewTopic ordersTopic() {
    return TopicBuilder.name("orders.events")
        .partitions(6)
        .replicas(3)
        .config(TopicConfig.RETENTION_MS_CONFIG, "604800000") // 7 days
        .config(TopicConfig.CLEANUP_POLICY_CONFIG, "delete")
        .build();
  }

  @Bean
  public NewTopic ordersDltTopic() {
    return TopicBuilder.name("orders.events.DLT").partitions(3).replicas(3).build();
  }
}
KEY POINTS
ENABLE_IDEMPOTENCE
Exactly-once per partitionPrevents duplicate records on producer retry. Requires acks=all and retries > 0. Spring auto-sets these when idempotence is enabled.
TopicBuilder
Auto-create topics on startupDeclare NewTopic beans and Spring Kafka creates them via the Admin client. Idempotent — safe to run on every restart.
JsonSerializer
Auto-serialize POJOsSpring's JsonSerializer uses Jackson to convert your model classes to JSON bytes. Set ADD_TYPE_INFO_HEADERS=false to avoid tight coupling between producer and consumer schemas.
LINGER_MS + BATCH_SIZE
Throughput tuningWait up to 10ms to fill a 64KB batch before sending. More records per network request = better throughput and compression ratio.

@Configuration
@EnableKafka
public class KafkaConsumerConfig {

  @Value("${spring.kafka.bootstrap-servers}")
  private String bootstrapServers;

  @Bean
  public ConsumerFactory<String, OrderEvent> consumerFactory() {
    JsonDeserializer<OrderEvent> deser = new JsonDeserializer<>(OrderEvent.class);
    deser.addTrustedPackages("com.example.*");
    deser.setUseTypeMapperForKey(false);

    Map<String, Object> props = new HashMap<>();
    props.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG,        bootstrapServers);
    props.put(ConsumerConfig.GROUP_ID_CONFIG,                 "order-processor-v2");
    props.put(ConsumerConfig.AUTO_OFFSET_RESET_CONFIG,        "earliest");
    props.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG,        false);
    props.put(ConsumerConfig.ISOLATION_LEVEL_CONFIG,           "read_committed");
    props.put(ConsumerConfig.MAX_POLL_RECORDS_CONFIG,          500);
    props.put(ConsumerConfig.SESSION_TIMEOUT_MS_CONFIG,         45000);
    props.put(ConsumerConfig.HEARTBEAT_INTERVAL_MS_CONFIG,     15000);
    props.put(ConsumerConfig.PARTITION_ASSIGNMENT_STRATEGY_CONFIG,
        List.of(CooperativeStickyAssignor.class));
    return new DefaultKafkaConsumerFactory<>(props,
        new StringDeserializer(), deser);
  }

  // ── Concurrent listener container factory ─────
  @Bean
  public ConcurrentKafkaListenerContainerFactory<String, OrderEvent>
      kafkaListenerContainerFactory(
          DefaultErrorHandler errorHandler) {

    var factory = new ConcurrentKafkaListenerContainerFactory<String, OrderEvent>();
    factory.setConsumerFactory(consumerFactory());
    factory.setConcurrency(3);
    factory.getContainerProperties().setAckMode(ContainerProperties.AckMode.MANUAL_IMMEDIATE);
    factory.setCommonErrorHandler(errorHandler);
    factory.setBatchListener(false);  // true for batch mode
    return factory;
  }
}
KEY POINTS
MANUAL_IMMEDIATE
Ack modeOffset is committed the moment acknowledgment.acknowledge() is called in your listener. Never commit before processing — use this with enable-auto-commit: false.
setConcurrency(3)
Parallel listener threadsCreates 3 consumer threads per listener. Each thread owns partitions from the group. Should not exceed the number of partitions — extra threads sit idle.
CooperativeStickyAssignor
Zero-downtime rebalancesOnly reassigns partitions that need to move. Consumers keep their existing partitions during rebalance, eliminating the stop-the-world pause of EagerAssignor.
read_committed
Skip dirty readsConsumers only see records from committed transactions. Required for exactly-once end-to-end when paired with a transactional producer.
OrderEventProducer.java
KAFKA
@Service
@Slf4j
@RequiredArgsConstructor
public class OrderEventProducer {

  private final KafkaTemplate<String, Object> kafkaTemplate;

  @Value("${app.kafka.topics.orders}")
  private String ordersTopic;

  // ── Simple async send ──────────────────────────
  public CompletableFuture<SendResult<String, Object>>
      sendOrderEvent(OrderEvent event) {

    ProducerRecord<String, Object> record = new ProducerRecord<>(
        ordersTopic,
        event.getOrderId(),    // key → partition routing
        event
    );

    // Add custom headers for tracing
    record.headers().add("X-Correlation-Id",
        UUID.randomUUID().toString().getBytes());
    record.headers().add("X-Source-Service", "order-service".getBytes());

    return kafkaTemplate.send(record)
        .whenComplete((result, ex) -> {
          if (ex != null) {
            log.error("Failed to send order event: {}",
                event.getOrderId(), ex);
          } else {
            log.debug("Sent [{}] to {}@{}",
                event.getOrderId(),
                result.getRecordMetadata().topic(),
                result.getRecordMetadata().offset());
          }
        });
  }

  // ── Batch send ────────────────────────────────
  public void sendBatch(List<OrderEvent> events) {
    events.forEach(e -> sendOrderEvent(e));
    kafkaTemplate.flush();  // force immediate dispatch
  }
}
KEY POINTS
ProducerRecord key
Partition routingRecords with the same key always go to the same partition, guaranteeing ordering for that key. Use the entity ID (orderId, userId) as the key.
whenComplete()
Non-blocking callbacksReturns a CompletableFuture — never call .get() in a hot path. Use whenComplete for logging/metrics and let callers chain futures as needed.
record.headers()
Metadata propagationUse headers to carry correlation IDs, source service names, schema version, or trace context (W3C Trace-Context). Consumers can read these without deserializing the body.
flush()
Force batch sendWithout flush, records sit in the buffer until linger.ms expires. Call flush() after a batch to dispatch immediately — useful at the end of a bulk import job.

@Component
@Slf4j
@RequiredArgsConstructor
public class OrderEventConsumer {

  private final OrderService orderService;

  // ── Single-record listener ─────────────────────
  @KafkaListener(
      topics   = "${app.kafka.topics.orders}",
      groupId  = "order-processor-v2",
      containerFactory = "kafkaListenerContainerFactory"
  )
  public void onOrderEvent(
      @Payload          OrderEvent       event,
      @Header(KafkaHeaders.RECEIVED_TOPIC)     String topic,
      @Header(KafkaHeaders.RECEIVED_PARTITION) int    partition,
      @Header(KafkaHeaders.OFFSET)              long   offset,
      Acknowledgment                                    ack
  ) {
    log.info("Consuming [{}] from {}:{}@{}",
        event.getOrderId(), topic, partition, offset);

    orderService.process(event);

    ack.acknowledge();  // commit offset AFTER processing
  }

  // ── Batch listener (high-throughput) ───────────
  @KafkaListener(
      topics   = "${app.kafka.topics.payments}",
      groupId  = "payment-processor",
      containerFactory = "batchListenerContainerFactory"
  )
  public void onPaymentEventBatch(
      List<ConsumerRecord<String, PaymentEvent>> records,
      Acknowledgment ack
  ) {
    log.info("Batch of {} payment events", records.size());
    List<PaymentEvent> events = records.stream()
        .map(ConsumerRecord::value).toList();
    orderService.processBatch(events);
    ack.acknowledge();
  }

  // ── DLT listener (dead letter topic) ──────────
  @KafkaListener(topics = "${app.kafka.topics.orders-dlt}",
                 groupId = "orders-dlt-handler")
  public void onDeadLetter(ConsumerRecord<String, Object> record) {
    log.error("DLT record key={} value={}",
        record.key(), record.value());
    // alert, store in DB, trigger manual review...
  }
}
KEY POINTS
@Payload
Deserialized POJOSpring deserializes the Kafka value bytes into your model class automatically. Use @Header to inject metadata like topic, partition, and offset without parsing them manually.
ack.acknowledge()
Manual offset commitCall this only after your processing is complete and durable. If your service crashes before this line, the record will be redelivered — giving you at-least-once guarantees.
Batch listener
Bulk processingReceives up to max.poll.records at once. Ideal for DB bulk inserts or batch API calls. One ack commits all offsets in the batch.
DLT listener
Poison message handlingAfter max retry exhaustion, Spring Kafka routes the record to topic.DLT. Listen here to alert, persist failed records for analysis, or trigger manual review workflows.
KafkaErrorHandler.java
KAFKA
@Configuration
public class KafkaErrorHandler {

  @Bean
  public DefaultErrorHandler defaultErrorHandler(
      KafkaTemplate<String, Object> kafkaTemplate) {

    // ── Dead letter publisher ──────────────────────
    var dltPublisher = new DeadLetterPublishingRecoverer(kafkaTemplate,
        (record, ex) -> new TopicPartition(
            record.topic() + ".DLT",
            record.partition()  // same partition for ordering
        ));

    // ── Exponential backoff: 1s → 2s → 4s (3 tries) ─
    var backoff = ExponentialBackOffWithMaxRetries.builder()
        .initialInterval(1_000L)
        .multiplier(2.0)
        .maxRetries(3)
        .build();

    var handler = new DefaultErrorHandler(dltPublisher, backoff);

    // ── Don't retry on non-recoverable errors ──────
    handler.addNotRetryableExceptions(
        JsonProcessingException.class,   // bad JSON → DLT immediately
        ValidationException.class,       // business rule violation
        DeserializationException.class   // unreadable payload
    );

    // ── Log all failures ───────────────────────────
    handler.setRetryListeners((record, ex, deliveryAttempt) ->
        log.warn("Retry #{} for key={}", deliveryAttempt, record.key(), ex));

    return handler;
  }
}
KEY POINTS
DeadLetterPublishingRecoverer
Auto-publish to DLTAfter retries are exhausted, the failed record is published to topic.DLT. The resolver lambda lets you control which DLT partition to use — keeping it on the same partition preserves ordering for analysis.
ExponentialBackoff
Retry with backoffRetries at 1s, 2s, 4s before giving up. Prevents a flapping downstream from hammering the system. The consumer thread blocks during retries — for high-throughput, consider non-blocking retry with a retry topic.
addNotRetryableExceptions
Fast-fail non-transient errorsDeserialization failures and validation errors will never succeed on retry — send them straight to DLT. Only retry transient failures like DB timeouts or HTTP 503s.
OrderService.java — Transactional Producer
KAFKA
@Service
@Slf4j
@RequiredArgsConstructor
public class OrderService {

  private final KafkaTemplate<String, Object> kafka;
  private final OrderRepository                   repo;

  // ── Exactly-once: DB write + Kafka publish ─────
  // Both are atomic — if Kafka fails, DB rolls back
  @Transactional("chainedKafkaTransactionManager")
  public void createOrder(CreateOrderRequest req) {
    Order order = repo.save(Order.from(req));  // DB save

    OrderEvent event = OrderEvent.builder()
        .orderId(order.getId().toString())
        .status("CREATED")
        .timestamp(Instant.now())
        .build();

    kafka.send("orders.events", event.getOrderId(), event());
  }

  // ── Consume-transform-produce (read-process-write)
  @KafkaListener(topics = "orders.events")
  @Transactional("chainedKafkaTransactionManager")
  public void processAndEmit(OrderEvent in) {
    // Processing + outgoing publish are one atomic unit
    PaymentEvent payment = buildPayment(in);
    kafka.send("payments.events", payment.getId(), payment);
  }
}
KEY POINTS
@Transactional (chained)
DB + Kafka atomicityUse ChainedKafkaTransactionManager to bind your JPA transaction manager to Kafka's. If Kafka commit fails, the DB transaction rolls back — ensuring both are always in sync.
transaction-id-prefix
Producer fencingSet in application.yml. Each producer instance gets a unique transactional ID (prefix + counter). Brokers use this to fence zombie producers from the previous instance after a restart.
read-process-write
Exactly-once stream processingConsuming from one topic and producing to another in the same transaction ensures the output is published if and only if the input offset is committed — no duplicates, no data loss.

@Configuration
public class RabbitConfig {

  // ── Exchange (topic type) ─────────────────────
  @Bean
  public TopicExchange ordersExchange() {
    return ExchangeBuilder.topicExchange("orders.exchange")
        .durable(true)
        .withArgument("alternate-exchange", "orders.unrouted")
        .build();
  }

  // ── Dead Letter Exchange ──────────────────────
  @Bean
  public DirectExchange dlxExchange() {
    return ExchangeBuilder.directExchange("orders.dlx").durable(true).build();
  }

  // ── Quorum queue with DLX + TTL ───────────────
  @Bean
  public Queue emailQueue() {
    return QueueBuilder.durable("orders.email.queue")
        .quorum()                     // Raft-based HA queue
        .deadLetterExchange("orders.dlx")
        .deadLetterRoutingKey("failed.email")
        .ttl(86_400_000)              // 24h message TTL
        .maxLength(100_000L)            // max queue depth
        .overflow(Overflow.rejectPublishDlx)
        .withArgument("x-delivery-limit", 5) // quorum redelivery limit
        .build();
  }

  // ── Dead letter queue ─────────────────────────
  @Bean
  public Queue dlQueue() {
    return QueueBuilder.durable("orders.dead-letter").quorum().build();
  }

  // ── Binding ───────────────────────────────────
  @Bean
  public Binding emailBinding(Queue emailQueue, TopicExchange ordersExchange) {
    return BindingBuilder.bind(emailQueue)
        .to(ordersExchange)
        .with("order.#");  // matches order.created, order.shipped.eu
  }

  // ── RabbitTemplate with publisher confirms ────
  @Bean
  public RabbitTemplate rabbitTemplate(ConnectionFactory cf) {
    var tpl = new RabbitTemplate(cf);
    tpl.setMessageConverter(jackson2MessageConverter());
    tpl.setMandatory(true);  // trigger returns on unroutable

    // Confirm callback
    tpl.setConfirmCallback((cd, ack, cause) -> {
      if (!ack) log.error("Broker NACK: {}", cause);
    });

    // Return callback (unroutable messages)
    tpl.setReturnsCallback(msg ->
        log.warn("Returned: replyCode={} exchange={} routingKey={}",
            msg.getReplyCode(), msg.getExchange(), msg.getRoutingKey()));
    return tpl;
  }

  @Bean
  public Jackson2JsonMessageConverter jackson2MessageConverter() {
    return new Jackson2JsonMessageConverter();
  }
}
KEY POINTS
QueueBuilder.quorum()
Raft-based HAQuorum queues replicate via Raft consensus across cluster nodes — much safer than classic mirrored queues. Always use for production workloads. Requires RabbitMQ 3.8+.
alternate-exchange
Unroutable message safety netIf a message matches no binding, it's forwarded to this exchange instead of silently dropped. Wire a logging queue to it to detect routing misconfigurations early.
publisher-confirm-type
Delivery confirmationCORRELATED mode lets you correlate each ACK/NACK back to the original message via a CorrelationData ID. Use for at-least-once guarantees on the publish side.
x-delivery-limit
Poison message protectionQuorum queues track redelivery count. After this limit, the message is automatically dead-lettered. Prevents a bad message from blocking consumer threads indefinitely.
OrderPublisher.java

@Service
@Slf4j
@RequiredArgsConstructor
public class OrderPublisher {

  private final RabbitTemplate rabbitTemplate;

  @Value("${app.rabbit.exchange}")
  private String exchange;

  // ── Fire-and-forget with publisher confirm ─────
  public void publishOrderCreated(OrderEvent event) {
    String routingKey = "order.created." + event.getRegion();

    CorrelationData cd = new CorrelationData(event.getOrderId());

    rabbitTemplate.convertAndSend(
        exchange,
        routingKey,
        event,
        cd
    );

    // Async confirm result
    cd.getFuture().thenAccept(confirm -> {
      if (!confirm.isAck())
        log.error("Broker NACK for order {}", event.getOrderId());
      else
        log.debug("ACK for order {}", event.getOrderId());
    });
  }

  // ── Message with custom properties ────────────
  public void publishWithExpiry(OrderEvent event, Duration ttl) {
    rabbitTemplate.convertAndSend(exchange, "order.created", event,
        msg -> {
          MessageProperties props = msg.getMessageProperties();
          props.setExpiration(String.valueOf(ttl.toMillis()));
          props.setDeliveryMode(MessageDeliveryMode.PERSISTENT);
          props.setPriority(5);
          props.setCorrelationId(event.getOrderId());
          props.setHeader("X-Source", "order-service");
          return msg;
        });
  }
}
KEY POINTS
CorrelationData
Track individual confirmsPass a unique ID per message. The confirm callback on RabbitTemplate receives it back so you can match the broker's ACK/NACK to the specific message that was published.
routing key pattern
Dynamic routing"order.created." + region produces keys like order.created.eu. Queues bound with order.# receive all of these; a queue bound with order.created.eu receives only European orders.
MessageDeliveryMode.PERSISTENT
Survive broker restartSet on every message destined for a durable queue. Without this, messages are stored in RAM only and lost on broker restart even if the queue is durable.
setExpiration()
Per-message TTLOverride the queue-level TTL for individual messages. Useful for time-sensitive notifications that should be discarded (and dead-lettered) if not consumed within a tight window.
OrderConsumer.java

@Component
@Slf4j
@RequiredArgsConstructor
public class OrderConsumer {

  private final EmailService emailService;

  // ── Standard listener ─────────────────────────
  @RabbitListener(
      queues           = "${app.rabbit.queue}",
      ackMode          = "MANUAL",
      concurrency      = "3-10"         // min-max threads
  )
  public void onOrderEvent(
      @Payload  OrderEvent      event,
      @Header(AmqpHeaders.DELIVERY_TAG) long deliveryTag,
      @Header(AmqpHeaders.REDELIVERED)  boolean redelivered,
      Channel                           channel
  ) throws IOException {

    log.info("Processing order {} (redelivered={})",
        event.getOrderId(), redelivered);

    try {
      emailService.sendConfirmation(event);
      channel.basicAck(deliveryTag, false);  // ✅ success

    } catch (TransientException e) {
      // Requeue once; if already redelivered → DLX
      channel.basicNack(deliveryTag, false, !redelivered);

    } catch (Exception e) {
      log.error("Fatal error, sending to DLX", e);
      channel.basicNack(deliveryTag, false, false);  // → DLX
    }
  }

  // ── Batch listener ────────────────────────────
  @RabbitListener(queues = "orders.batch.queue", ackMode = "MANUAL")
  public void onBatch(
      List<Message> messages,
      Channel channel
  ) throws IOException {
    log.info("Batch of {} messages", messages.size());
    // process all...
    long lastTag = Long.parseLong(
        messages.getLast().getMessageProperties().getDeliveryTag().toString());
    channel.basicAck(lastTag, true);  // ACK all up to lastTag
  }
}
KEY POINTS
basicAck / basicNack
Manual AMQP acknowledgmentAlways use MANUAL ack mode in production. ACK removes the message from the queue; NACK with requeue=true puts it back at the front; NACK with requeue=false forwards it to the DLX.
redelivered flag
Detect retriesRabbitMQ sets this to true on the second delivery attempt. Use it to avoid infinite requeue loops — if the message is already a redeliver and still fails, send it to DLX immediately.
concurrency = "3-10"
Auto-scaling threadsSpring AMQP starts 3 consumer threads and scales up to 10 based on queue depth. More efficient than fixed concurrency when load is bursty.
basicAck(tag, multiple=true)
Batch ACKACKs all unacknowledged messages up to and including the given delivery tag in a single AMQP frame. Reduces network round-trips for batch processing significantly.

/**
 * Delayed retry pattern using DLX + per-message TTL.

@Configuration
public class RetryTopology {

  // Retry queue: hold for 30s, then re-route to main exchange
  @Bean
  public Queue retryQueue() {
    return QueueBuilder.durable("orders.retry.30s")
        .ttl(30_000)                        // hold for 30s
        .deadLetterExchange("orders.exchange")  // re-enter main exchange
        .deadLetterRoutingKey("order.#")
        .build();
  }
}

@Component
@Slf4j
@RequiredArgsConstructor
public class DlxConsumer {

  private final RabbitTemplate   rabbitTemplate;
  private final AlertService     alertService;
  private final FailedMsgRepo    failedMsgRepo;

  @RabbitListener(queues = "orders.dead-letter", ackMode = "MANUAL")
  public void onDeadLetter(
      Message  message,
      Channel  channel
  ) throws IOException {

    MessageProperties props = message.getMessageProperties();

    // Extract death metadata
    List<Map<String,?>> xDeath = (List<?>)
        props.getHeaders().get("x-death");

    long deathCount = (Long) xDeath.get(0).get("count");
    String reason    = (String) xDeath.get(0).get("reason");

    log.error("Dead-lettered #{} reason={} correlationId={}",
        deathCount, reason, props.getCorrelationId());

    // Persist + alert for human review
    failedMsgRepo.save(FailedMessage.from(message));
    alertService.notifyOnCall(reason, deathCount);

    channel.basicAck(props.getDeliveryTag(), false);
  }
}
KEY POINTS
Retry queue pattern
Delayed retry without pluginsNACKed messages go to a TTL-only queue. After TTL expires, they dead-letter back to the main exchange — effectively a 30-second retry delay with no external scheduler needed.
x-death header
Death metadataRabbitMQ automatically appends death metadata to dead-lettered messages — including the original queue, exchange, routing key, reason, and how many times the message has been dead-lettered. Use this for audit trails.
failedMsgRepo.save()
Persist failuresAlways persist DLX messages to a database before ACKing them. This gives you an audit trail and enables manual reprocessing tooling without losing the message payload.
ACK on DLX queue
Don't forget to ACKThe DLX queue is just a normal queue — messages must still be ACKed by your DLX consumer. If you don't ACK, unprocessed messages pile up and block new dead-lettered arrivals.
