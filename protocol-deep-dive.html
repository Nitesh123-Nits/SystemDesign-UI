REST

Representational State Transfer — the backbone of the modern web. Stateless, resource-based HTTP communication between client and server.

HTTP/1.1 · HTTP/2 Stateless JSON / XML Speed: Medium

Request / Response Flow

How It Works

REST uses standard HTTP verbs (GET, POST, PUT, DELETE, PATCH) to perform CRUD operations on resources identified by URLs. Each request is completely stateless — the server stores no session info. Responses are typically JSON or XML. The client sends a request and waits for a complete response before continuing.

Code Example

// Fetch user data via REST const res = await fetch('/api/users/42', { method: 'GET', headers: { 'Authorization': 'Bearer token' } }); const user = await res.json(); // → { id: 42, name: "Alice" }

Pros

Universal — every platform supports HTTP natively
Stateless design scales horizontally with ease
Excellent caching via HTTP cache headers
Huge ecosystem of tools (Swagger, Postman, etc.)

Cons

Over-fetching or under-fetching data is common
Multiple round-trips needed for related resources
No real-time push — client must poll for updates

Best Use Cases

Public APIs CRUD backends Mobile apps Microservices E-commerce Auth services

Rule of thumb: If you're building a CRUD API consumed by multiple clients, REST is the default choice. It's well-understood, widely supported, and easy to debug.

GraphQL

🔍

GraphQL

A query language for APIs — ask for exactly what you need, nothing more. Developed by Meta in 2012, open-sourced in 2015.

HTTP POST Single Endpoint Typed Schema Speed: Medium

Query Resolution — Fetch Only What You Need

How It Works

GraphQL exposes a single endpoint (typically /graphql). The client sends a query describing exactly which fields it needs. The server's resolvers fetch data from any source (DB, REST, cache) and return only the requested fields. Supports queries (read), mutations (write), and subscriptions (real-time).

Code Example

// GraphQL query — ask for exactly what you need const QUERY = ` query GetUser($id: ID!) { user(id: $id) { name email posts { title } } } `; await fetch('/graphql', { method: 'POST', body: JSON.stringify({ query: QUERY, variables: { id: '42' } }) });

Pros & Cons

No over-fetching — get exactly what you asked for
Self-documenting via introspection & typed schema
Aggregate multiple data sources in one request
Complex caching (no URL-based HTTP caching)
N+1 query problem without DataLoader pattern
Learning curve for teams new to it

Best Use Cases

Complex frontends Mobile apps BFF layer Data-heavy dashboards Federated APIs

Rule of thumb: Use GraphQL when different clients (mobile vs web) need different shapes of the same data, or when you're aggregating many microservices.

Webhooks

Webhooks

Event-driven HTTP callbacks — the server pushes data to your URL when something happens, eliminating the need to poll.

HTTP POST Event-driven Fire & Forget Speed: Fast

Event Push — No Polling Required

How It Works

You register a callback URL with a third-party service. When an event occurs (payment completed, PR merged, etc.), the service sends an HTTP POST to your URL with the event payload. Your server responds with 200 OK to acknowledge. Unlike REST, no polling — the service comes to you. Security is enforced via HMAC signature headers.

Code Example

// Express webhook handler app.post('/webhook/stripe', (req, res) => { const sig = req.headers['stripe-signature']; const event = stripe.webhooks.constructEvent( req.body, sig, process.env.SECRET ); if (event.type === 'payment_intent.succeeded') { fulfillOrder(event.data.object); } res.json({ received: true }); // ACK });

Pros & Cons

Zero polling overhead — events arrive instantly
Simple integration with any HTTP server
Automatic retries from the provider on failure
Requires a publicly accessible endpoint
Hard to debug locally (need tunnels like ngrok)
Must handle duplicate delivery idempotently

Best Use Cases

Payment events CI/CD triggers Form submissions E-commerce orders Slack bots Zapier / n8n flows
WebSocket

🔁

WebSocket

Full-duplex persistent connection over a single TCP socket. Both client and server can send messages at any time — true real-time communication.

TCP Bidirectional Persistent Speed: Fast

Persistent Bidirectional Channel

How It Works

WebSocket starts with an HTTP handshake (Upgrade request), then upgrades to a persistent TCP connection. Once connected, either side can send frames at any time with no request-response overhead. The connection stays open until either side closes it — making it ideal for live collaboration, gaming, and real-time dashboards.

Code Example

// Server (Node.js / ws library) const wss = new WebSocketServer({ port: 8080 }); wss.on('connection', (ws) => { ws.on('message', (msg) => { // Broadcast to all clients wss.clients.forEach(c => c.send(msg)); }); }); // Client const ws = new WebSocket('wss://api.example.com'); ws.onmessage = (e) => render(e.data); ws.send(JSON.stringify({ type: 'chat', text }));

Pros & Cons

True real-time — sub-millisecond message delivery
Low overhead after handshake (2–10 byte frames)
Full-duplex — server pushes without polling
Stateful — harder to scale horizontally
Firewalls / proxies may interfere with long connections
Reconnection logic must be implemented manually

Best Use Cases

Live chat Multiplayer games Live dashboards Collaborative editing Trading platforms Live sports scores

gRPC

Google's high-performance RPC framework using Protocol Buffers over HTTP/2. The standard for microservice-to-microservice communication.

HTTP/2 Protocol Buffers Bi-streaming Speed: Very Fast

Binary Protocol Over HTTP/2 Multiplexed Streams

How It Works

Define your service contract in a .proto file. The protoc compiler generates type-safe client/server stubs in any language. Messages are serialized to compact binary (Protobuf) — ~5–10× smaller than JSON. Transported over HTTP/2 which provides multiplexing, streaming, and header compression. Supports 4 call types: unary, server-streaming, client-streaming, and bidirectional.

Code Example

// user.proto service UserService { rpc GetUser(UserRequest) returns (User); rpc StreamUsers(Filter) returns stream User; } // Node.js client call const stub = new UserServiceClient(addr, creds); stub.getUser({ id: 42 }, (err, user) => { console.log(user.name); // binary deserialized });

Pros & Cons

Extremely fast — binary serialization, HTTP/2 multiplexing
Strongly typed — compile-time contract enforcement
Polyglot — generates code for 10+ languages
Built-in streaming support (4 modes)
Not human-readable (binary) — harder to debug
Limited browser support without gRPC-Web proxy
Schema changes require proto recompilation

Best Use Cases

Microservices mesh Real-time streaming Mobile backends ML model serving Low-latency APIs

MQTT

Message Queuing Telemetry Transport — a lightweight pub/sub protocol designed for constrained devices and unreliable networks.

TCP Pub / Sub 2-byte header Speed: Lightweight

Publish / Subscribe via Central Broker

How It Works

Devices (publishers) send messages to topics on a broker (e.g., sensors/temperature). Other devices (subscribers) tell the broker which topics they care about using wildcards (sensors/#). The broker routes messages. With a 2-byte minimum header, MQTT is ideal for microcontrollers with limited RAM and bandwidth. Supports 3 QoS levels (0, 1, 2).

Code Example

// Node.js MQTT publisher (IoT device) const client = mqtt.connect('mqtt://broker.io'); client.on('connect', () => { client.publish( 'sensors/temperature', JSON.stringify({ val: 23.4, unit: 'C' }), { qos: 1 } // at-least-once delivery ); }); // Subscriber (dashboard) client.subscribe('sensors/#'); client.on('message', (topic, msg) => updateDashboard(topic, parse(msg)) );

Pros & Cons

Ultra-lightweight — 2-byte header, ideal for IoT
Decoupled — publishers & subscribers don't know each other
QoS levels for guaranteed delivery
Central broker is a single point of failure
No built-in message queuing (use broker features)
Binary topics — no native HTTP semantics

Best Use Cases

IoT sensor networks Smart home devices Industrial SCADA Vehicle telemetry Wearables
AMQP
07
📬

AMQP

Advanced Message Queuing Protocol — enterprise-grade reliable messaging with routing, acknowledgments, and durable queues.

TCP Message Queue Durable Speed: Reliable

Exchange → Queue → Consumer Routing

How It Works

Producers send messages to an Exchange with a routing key. The exchange routes based on type (direct, topic, fanout, headers) to one or more Queues. Consumers pull from queues and send ACK/NACK to confirm processing. Messages are durably persisted to disk. If a consumer fails, the broker re-queues the message. RabbitMQ is the most popular AMQP 0-9-1 implementation.

Code Example

// RabbitMQ with amqplib const ch = await conn.createChannel(); await ch.assertQueue('orders', { durable: true }); // Producer ch.sendToQueue('orders', Buffer.from(JSON.stringify(order)), { persistent: true } ); // Consumer (with manual ACK) ch.consume('orders', async (msg) => { await processOrder(JSON.parse(msg.content)); ch.ack(msg); // remove from queue });

Pros & Cons

Guaranteed delivery with ACK/NACK mechanism
Flexible routing via exchange types
Message persistence survives broker restart
Higher infrastructure overhead than MQTT
More complex setup vs. REST webhooks
Not suitable for real-time interactive apps

Best Use Cases

Order processing Task queues Email / SMS dispatch Background jobs Saga orchestration Audit logging

SSE

Server-Sent Events — a simple, lightweight one-way stream from server to client over regular HTTP. Perfect for live updates and notifications.

HTTP/1.1+ text/event-stream Auto-reconnect Speed: Lightweight

Unidirectional Server-to-Client Stream

How It Works

The client opens a single HTTP connection with Accept: text/event-stream. The server keeps the connection open and streams newline-delimited text events. Each event has an optional id, event type, and data field. The browser's built-in EventSource API handles automatic reconnection using the last event ID. Unlike WebSockets, it runs over plain HTTP — no upgrade needed.

Code Example

// Express SSE endpoint app.get('/events', (req, res) => { res.setHeader('Content-Type', 'text/event-stream'); res.setHeader('Cache-Control', 'no-cache'); const send = (data) => res.write(`data: ${JSON.stringify(data)}\n\n`); const iv = setInterval(() => send({ price: getLatestPrice() }), 1000 ); req.on('close', () => clearInterval(iv)); }); // Client const es = new EventSource('/events'); es.onmessage = (e) => updatePrice(JSON.parse(e.data));

SSE vs WebSocket

Works over HTTP — no firewall issues, proxy-friendly
Automatic reconnection built into EventSource
Much simpler than WebSocket for one-way data
Server → Client only (use fetch for client→server)
HTTP/1.1 limit of 6 connections per browser
Text-only (no binary frames like WebSocket)

Best Use Cases

Live notifications News feeds Stock tickers AI token streaming Progress bars Log tailing

Note: This is how LLM APIs (like Claude) stream tokens back to the browser — SSE is perfect for progressive text generation.

SOAP

Simple Object Access Protocol — a heavyweight, XML-based protocol with strict contracts via WSDL. The enterprise standard before REST took over.

HTTP · SMTP · TCP XML / WSDL WS-Security Speed: Heavy

SOAP Envelope Structure

How It Works

Every SOAP message is wrapped in an XML Envelope containing a Header (authentication, routing) and Body (the actual operation). The service is described in a WSDL (Web Services Description Language) file which clients use to auto-generate typed proxies. SOAP supports WS-Security for enterprise-grade security, transactions (WS-AtomicTransaction), and reliable messaging.

Code Example

/* SOAP request envelope */ <?xml version="1.0"?> <soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/..."> <soap:Header> <auth><token>abc123</token></auth> </soap:Header> <soap:Body> <GetUser xmlns="http://example.com"> <UserId>42</UserId> </GetUser> </soap:Body> </soap:Envelope>

Pros & Cons

Strict contracts prevent integration mismatches
WS-Security for enterprise compliance (HIPAA, PCI)
Works over HTTP, SMTP, TCP — transport agnostic
Extremely verbose XML — high bandwidth overhead
Complex tooling required (WSDL parsers, code gen)
REST is simpler for 95% of modern use cases

When You Still Need It

Banking systems Healthcare (HL7) Government portals Legacy ERP (SAP) Payment gateways

Modern reality: Avoid SOAP for new projects. Use it only when integrating with legacy enterprise systems that mandate it.
