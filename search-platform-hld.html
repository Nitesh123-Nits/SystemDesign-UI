⚡ Eventual Consistency (Acceptable)
Search is inherently eventually consistent. A newly published page may take minutes-to-hours to be crawled, indexed, and served. This is acceptable for general web search. The serving layer uses the latest available index shard and does NOT require strong consistency guarantees.

// 05 — PLANNING
Capacity
Planning
Growth planning, scaling strategies, and resource allocation across the system lifecycle.
Growth Phases
Phase 1 — MVP (1B pages, 1K QPS)
Single-region deployment. Monolithic crawler with 50 nodes. Single inverted index cluster (~50 shards). Basic BM25 ranking. No ML re-ranking. Simple LRU cache. Target: prove correctness and basic functionality.
Phase 2 — Scale-Out (10B pages, 10K QPS)
Shard index by URL hash. Add distributed crawl scheduler (Kafka-based URL frontier). Introduce ML ranking (offline-trained GBDT). Add Redis result cache. Deploy CDN for serving. Introduce query serving tier with load balancer.
Phase 3 — Production (50B pages, 100K QPS)
Multi-region active-active. 500+ index shards with 3× replication. Two-phase ranking: Retrieval (BM25) → Neural re-ranking (BERT/LLM). Streaming freshness pipeline. Full personalization. Autocomplete service. Knowledge Graph integration. A/B testing at serving layer.
Index Sharding Strategy
🔢 Document-Partitioned Index
Shard by docID range or URL hash. Each shard holds a complete local inverted index for its documents. Query fan-out: send query to all shards, merge top-K results.

Pros: Simple, easy rebalancing.
Cons: Fan-out to all shards on every query, network intensive.
📝 Term-Partitioned Index
Shard by term hash. Each shard owns all postings for a subset of terms. Query is decomposed into terms, each fetched from different shards.

Pros: Fewer shards hit per query.
Cons: Hot shards for popular terms (mitigation: replicate hot terms).

// 07 — COMPONENTS
Core
Components
Deep description of each major component, its responsibilities, technology choices, and internal design.
Crawler System
🕷 Web Crawler
  • URL Frontier: Priority queue (BFS + PageRank priority). Use Kafka as distributed queue with priority topics.
  • DNS Resolver: Local DNS cache to reduce lookup latency. Batch resolve URLs.
  • Fetcher: Async HTTP/2 with connection pooling. 100ms timeout.
  • Robots.txt Cache: Per-domain cache with 24h TTL.
  • Politeness: Rate limit 1 req/sec per domain. Respect crawl-delay header.
  • Deduplication: SimHash fingerprint stored in Redis. Reject near-duplicates.
⚙️ Document Processor
  • HTML Parser: Extract clean text, title, meta description, headings.
  • Link Extractor: Extract all outbound links, normalize URLs.
  • Language Detection: fastText langdetect on first 500 chars.
  • Content Quality: Filter thin content (<100 words), spam pages.
  • JS Rendering: Headless Chromium pool for JS-heavy pages (10% of pages).
  • Structured Data: Extract JSON-LD, OpenGraph, schema.org markup.
Index Builder
📑 Inverted Index Construction (MapReduce)
// Map Phase: emit (term, docID, tf, position) map(docID, document) { tokens = tokenize(document.text) for (pos, term in tokens): tf = count(term, document) emit(term, {docID, tf, pos, fieldWeight}) } // Reduce Phase: build posting list per term reduce(term, postings[]) { sorted = sortBy(postings, 'docID') // Delta-encode docIDs for compression compressed = varIntEncode(sorted.docIDs) write(term → {compressed_postings, df: len(postings)}) } // Posting List Format per term: // [docID_delta, tf, positions[], field_flags] // Compressed with VByte encoding → 4-8 bytes/entry
Query Processor Components
🔠 Query Parser
Tokenize, normalize case, handle operators (AND/OR/NOT/NEAR), expand synonyms, identify entities and intents.
✏️ Spell Corrector
Noisy channel model. Edit distance + language model probability. Uses query logs for high-confidence corrections.
🧠 Intent Classifier
Classify query intent: navigational, informational, transactional. Routes to appropriate vertical (web/images/maps).
🔀 Query Fan-out
Scatter query to all index shards in parallel. Collect top-K candidates from each shard. Merge and globally rank.
✂️ Snippet Generator
Extract best passage containing query terms. Highlight matches. Use forward index to fetch original text. Max 160 chars.
📐 Feature Extractor
Extract 200+ ranking features: TF-IDF, BM25, PageRank, freshness, click-through rate, anchor text, domain authority.

// 09 — DEEP DIVE
Crawling &
Indexing
Deep dive into the web crawler architecture and distributed inverted index construction.
URL Frontier Design
Priority Queue Architecture
Front queues (priority): Separate queues by URL priority score (PageRank + freshness + domain authority). High-priority URLs crawled more frequently.

Back queues (politeness): One queue per domain. Ensures only 1 outstanding request per domain at a time. Min delay enforced (1 req/sec default).

Implementation: Kafka topics per priority level. Back queues use per-domain state in Redis. Total frontier size: ~100B URLs, ~1.5TB Redis.
Duplicate Detection (SimHash)
URL Normalization: Lowercase, remove tracking params, canonical URL resolution.

Content SimHash: Compute 64-bit SimHash fingerprint of document. Store in Redis bitset. If Hamming distance <3 → near-duplicate, skip.

Exact duplicate: MD5 hash of stripped content. Reject if seen before.

Scale: 50B fingerprints × 8 bytes = 400GB in-memory fingerprint store.
Inverted Index Structure
// Inverted Index: term → posting list { "python": { df: 12_000_000, // document frequency postings: [ {docID: 4521, tf: 8, positions: [12,45,89], fields: "TITLE|BODY"}, {docID: 4522, tf: 3, positions: [5,110,234], fields: "BODY"}, // ... sorted by docID, delta-encoded ] } } // Compression: Variable-byte encoding // docID deltas: [4521, 1, 502, 3, ...] → 1-2 bytes each // Skip pointers every 128 entries → O(√n) skipping // Block-level compression (PForDelta) for posting lists >1000 // Intersection (AND query) using merge: function intersect(list1, list2): i, j = 0, 0 results = [] while i < len(list1) and j < len(list2): if list1[i].docID == list2[j].docID: results.append(list1[i]); i++; j++ elif list1[i].docID < list2[j].docID: i++ else: j++ return results // O(n+m) with skip pointers → O(√n + √m)
Freshness Pipeline
⚡ Real-Time Freshness (News & High-Priority)
RSS/Sitemap streaming: Monitor 100K+ news RSS feeds. Kafka producer ingests new URLs with HIGH priority flag.

Streaming indexer: Flink job processes high-priority documents through a mini-indexer. Updates a "fresh shard" overlay. Fresh shard results are merged with base index results at query time.

Freshness SLO: Breaking news indexed within 15 minutes. General web re-crawl every 2-30 days based on update frequency.

Query Understanding
🔤 Query Parsing Steps
  • Tokenization: Split on whitespace, handle hyphen/slash
  • Case folding: Lowercase (except acronyms)
  • Stop word removal: "the", "a", "is" (but NOT for phrase queries)
  • Stemming/Lemmatization: running → run (Porter Stemmer)
  • Operator detection: AND/OR/NOT/site:/filetype: operators
  • Quote handling: "exact phrase" → phrase query
  • Synonym expansion: car → automobile, car, vehicle
🧠 Query Intent Classification
  • Navigational: "youtube", "github login" → go to specific site
  • Informational: "how to make pasta" → provide information
  • Transactional: "buy iPhone 15" → intent to purchase
  • Local: "coffee shops nearby" → geo-bound results
  • Ambiguous: "jaguar" → car or animal? Use context
  • Question: "who is…", "what is…" → direct answer box
Autocomplete Architecture
⚡ Real-Time Autocomplete (<100ms)
DATA STRUCTURE
  • Trie (prefix tree) loaded in memory per datacenter
  • Each trie node stores top-10 completions by frequency
  • ~500GB in-memory trie for 1B+ query prefixes
  • Updated from query logs daily (offline batch)
  • Personalized layer: user's own history boosted
SERVING
  • Every keypress triggers request after 50ms debounce
  • Request to nearest datacenter (anycast)
  • Trie lookup: O(m) where m = prefix length
  • Trending queries boosted via real-time signal
  • SafeSearch filters out inappropriate suggestions

Index Sharding & Replication
🔀 Sharding Strategy
  • Document-partitioned: URL hash mod N shards (preferred)
  • Each shard: ~100M documents, ~64GB RAM for postings cache
  • Hot terms (e.g., "the") on every shard — use skip lists
  • Shard rebalancing via consistent hashing ring
  • 500 shards for 50B doc corpus
📦 Replication Strategy
  • 3× replication per shard (1 primary + 2 replicas)
  • Read from any replica (load balance reads)
  • Eventual consistency for index updates acceptable
  • Leader election via Raft for write coordination
  • Cross-region replication for disaster recovery
// 13 — DEEP DIVE
Caching
Strategy
Multi-level caching architecture from CDN to posting list cache to achieve sub-200ms P99 latency.
Cache Hierarchy
L0
CDN / Browser Cache
SERP pages cached at CDN edge for popular queries. Cache-Control: max-age=60. ~10% of traffic served here.
L1
Result Cache (Redis)
Top 30% queries cached. Key = normalized query string. TTL: 5 min general, 30 sec news. ~30% cache hit rate.
L2
Index Block Cache
OS page cache + custom buffer pool caches hot posting lists in RAM. Top 1% terms always hot (~80% of query volume).
Posting List Cache (Critical Path)
🔑 Zipf's Law: 1% of terms = 80% of queries
Query term distribution follows Zipf's law. The top 10,000 most frequent terms appear in 80% of all queries. Strategy: pre-load top 10K posting lists into RAM on every index shard. For 100M doc/shard, top posting lists = ~50GB per shard (fits in 64GB RAM). Cache eviction: LRU with frequency-based aging. Result: ~80% of inverted index lookups served from RAM → sub-10ms retrieval.

Hard Interview Questions & Answers
Q How would you handle a query for a very common term like "the" which appears in billions of documents? ▶
"The" is a stop word — we remove it from the index entirely for keyword queries. For phrase queries ("the matrix"), we keep positional information but use skip pointers to efficiently navigate huge posting lists. At the index shard level, the posting list for "the" is pre-cached in RAM (it's always hot), and we use SIMD instructions to vectorize the intersection. For BM25, high-IDF terms dominate scoring, so "the" contributes negligible score anyway.
Q How do you prevent a slow index shard from degrading P99 latency for all queries? ▶
This is the "straggler problem." Solutions: (1) Hedged requests — after 50ms, re-send the same request to a replica; use whichever responds first. (2) Speculative execution — proactively scatter to multiple replicas. (3) Timeout + partial results — if shard N doesn't respond in 150ms, return results from other shards only (graceful degradation). (4) Load shedding — circuit breaker marks overloaded shard as "slow" and routes to replicas. (5) Result from cache — stale results are better than timeout for most queries.
Q How would you design the freshness pipeline for news search? ▶
Separate real-time pipeline from the batch pipeline: (1) RSS/Sitemap poller monitors 100K+ news sources every 30 seconds. New URLs → Kafka high-priority topic. (2) Fast crawler pool dedicated to high-priority URLs, SLA: fetch within 2 minutes. (3) Streaming indexer (Flink) processes documents and writes to a news shard overlay — a small, fast-updating index separate from the main index. (4) At query time, the query processor hits BOTH the main index AND the news overlay. Results merged with freshness boost. News overlay is small (<1M docs) so it fits entirely in RAM.
Q How do you handle spam and SEO manipulation? ▶
Multi-layer defense: (1) TrustRank: start from known trusted seed sites (gov, edu) and propagate trust through link graph. Pages far from trusted seeds get penalized. (2) Spam classifier: ML model trained on labeled spam features: keyword stuffing, cloaking, link farms, thin content, ad-heavy pages. (3) Link quality: discount links from known link farms. Use link diversity (unique domains) not raw count. (4) Click signal validation: detect click fraud via bot detection (IP, user-agent, behavior). (5) Manual review: webmaster quality team reviews flagged sites. (6) Demotion on user feedback: high "back to search" rate demotes page.
Q How would you scale the system from 1K to 100K QPS? ▶
Horizontal scaling at each layer: (1) Query servers: Stateless — add more boxes, load balancer distributes. 1K QPS → 100 servers for 100K QPS. (2) Index shards: Add shards as corpus grows; add replicas as QPS grows. Each replica handles read traffic. (3) Cache: Result cache reduces effective QPS by 30-70%. Scale Redis cluster. (4) CDN: Popular SERP pages cached at edge. (5) ML inference: Add GPUs for neural re-ranking. (6) Data centers: Multi-region active-active — US, EU, APAC — reduces latency and spreads load. (7) Auto-scaling: K8s HPA scales query server pods on CPU/QPS metrics.
Q How do you compute PageRank at web scale (50B pages, 500B edges)? ▶
Use distributed graph processing: (1) Storage: Link graph stored in Bigtable/Colossus — row key = source URL hash, value = list of destination hashes. (2) Computation: Apache Spark GraphX or custom MapReduce. Each iteration: Map phase reads current PR values, emits (destination, contribution) pairs. Reduce phase sums contributions per destination. (3) Scale: ~50 iterations needed for convergence. At 500B edges × 16 bytes = 8TB. Runs on 10,000-core cluster in ~4 hours. (4) Incremental: Full recompute weekly. Daily incremental update for newly crawled pages using approximate local updates. (5) Personalized PR: Compute topic-sensitive PR (sports, tech, health) — 16 topic vectors, each with different teleportation distribution.
Q Should you use vector search (semantic) or inverted index (keyword)? Both? ▶
Modern search uses BOTH — this is called hybrid search: (1) Sparse retrieval (BM25/inverted index): Excellent for exact keyword matching, navigational queries, rare terms. Very fast. (2) Dense retrieval (vector search): Excellent for semantic similarity, paraphrases, zero-shot generalization. Requires ANN index (FAISS, ScaNN). Slower and more storage. (3) Hybrid: Run both in parallel, merge results using Reciprocal Rank Fusion (RRF) or learned fusion. The neural re-ranker (L3) is itself a semantic component that re-ranks candidate retrieved via BM25. Trade-off: vector search requires embedding inference at query time (adds ~10-20ms) and embedding store (~16KB per doc × 50B = 800TB).

Core Algorithms
📊 BM25 Formula
// BM25 score(D, Q) = Σ IDF(qi) × TF_norm // IDF(qi) = log((N - df_i + 0.5) / (df_i + 0.5)) // TF_norm = (tf × (k1+1)) / (tf + k1×(1-b+b×|D|/avgdl)) // k1 = 1.5 (term freq saturation), b = 0.75 (length norm) // N = corpus size, df = doc frequency, tf = term freq
🔗 PageRank Formula
// PR(A) = (1-d) + d × Σ PR(T_i) / C(T_i) // d = 0.85 (damping factor / teleportation) // T_i = pages linking to A // C(T_i) = number of outbound links from T_i // Converges in ~50 iterations for most corpora
Must-Know Data Structures
📝 Inverted Index
Maps term → posting list {docID, TF, positions}. Core of all search engines. Compressed with VByte/PForDelta. Skip pointers for fast intersection.
O(|q|) lookup
🌳 Trie (Prefix Tree)
Used for autocomplete. Each node = character. Walk prefix, collect candidates from subtree. O(m) lookup where m = prefix length. Memory-optimized with compressed trie.
O(m) prefix
🔢 Bloom Filter
Probabilistic set membership. Used in crawlers to check if URL was already seen. Space-efficient: 10 bits/element for 1% false positive rate. Never miss a URL (no false negatives).
O(k) ops
🧮 SimHash
Locality-sensitive hash for near-duplicate detection. Similar documents → similar 64-bit hashes. Hamming distance <3 → near-duplicate. Enables dedup at billions of pages.
64-bit LSH
🔵 Min-Hash / LSH
Jaccard similarity estimation via min-wise hashing. Approximate nearest neighbor search for vector embeddings (FAISS, ScaNN). Essential for semantic search at scale.
ANN search
📦 Skip Lists
Used in posting list traversal. Skip pointers every √n entries allow O(√n) skipping vs O(n) linear scan. Critical for AND queries on large posting lists.
O(√n) skip

// 16 — TRADE-OFFS
Design
Trade-offs
Every design decision involves trade-offs. Knowing these shows senior engineering thinking.
Key Design Trade-offs
⚡ Freshness vs. Coverage
Crawl more = better coverage but more resources and may miss updates.
Crawl faster = fresher but wastes budget on unchanged pages.

Solution: Adaptive crawl frequency — crawl frequently updated pages (news) every 15min, evergreen content every 30 days. Use Last-Modified and ETag headers. Monitor for sitemaps.
🎯 Precision vs. Recall
High precision = fewer results but very relevant. Better for most queries.
High recall = more results but more noise. Better for rare queries.

Solution: Two-phase retrieval. L1 optimizes recall (BM25, top-1000). L2/L3 optimize precision (ranking, re-ranking). User feedback (clicks) continuously calibrates the balance.
💰 Cost vs. Latency
Neural re-ranking (BERT): Best quality, costs GPU, adds 40ms.
GBDT ranking: Good quality, CPU-only, adds 10ms.
BM25 only: Cheapest, fastest, lowest quality.

Solution: Use ML re-ranking only for queries where it helps (ambiguous, head queries). Skip for navigational queries where first result is obvious.
🗄 Index Freshness vs. Stability
Streaming index updates: Always fresh, but complex, risk of inconsistency.
Batch index rebuilds: Stable and consistent, but hours of lag.

Solution: Hybrid. Main index rebuilt in weekly batch (stable, high-quality). News overlay updated in real-time (Flink). Merged at query time with freshness boost.
🔀 Document vs. Term Partitioning
Document-partitioned: Each shard has all terms for subset of docs. Query fans out to ALL shards. Simple, good load distribution.
Term-partitioned: Each shard has all docs for subset of terms. Query goes to fewer shards. But hot shards for popular terms.

Winner: Document-partitioned (Google uses this). Fan-out to all shards is manageable with hedged requests.
👤 Personalization vs. Privacy
More personalization = better relevance for user, but requires storing browsing history.
No personalization = better privacy, same results for everyone.

Solution: Opt-in personalization. Minimal server-side storage (session-based). Use on-device models for sensitive signals. Anonymize logs after 6 months. GDPR right-to-delete.
