STEP 01

Produce

Producer sends a keyed message to a topic. Key determines partition assignment (sticky routing).

STEP 02

Partition & Replicate

Message appended to partition log. Leader replicates to follower brokers. ZooKeeper/KRaft manages metadata.

STEP 03

Poll by Offset

Consumers in a group each own partitions and pull messages, tracking their own offset in __consumer_offsets.

STEP 04

Retain & Replay

Messages persist on disk per retention policy. Multiple consumer groups can read same data independently.

STEP 01

Publish

Publisher sends message to an Exchange with a routing key (e.g., "order.created") via AMQP protocol.

STEP 02

Route

Exchange evaluates routing key against bindings and delivers to matching queues. Fanout sends to all bound queues.

STEP 03

Queue & Push

Messages sit in durable queue. Broker pushes to consumers (prefetch controlled), not pulled like Kafka.

STEP 04

ACK & Delete

Consumer processes and sends ACK. Message is deleted from queue. NACK routes to Dead Letter Exchange (DLX).

04 — WHEN TO CHOOSE
?

Choose Kafka

Event Streaming Platform
  • You need to process millions of events per second (clickstream, IoT telemetry)
  • Multiple independent services need to consume the same events (fan-out reads)
  • You need event replay, time-travel debugging, or reprocessing historical data
  • Building event-driven microservices with full audit trail requirements
  • Real-time analytics pipelines (Kafka + Flink/Spark Streaming)
  • CQRS / Event Sourcing — the log IS the source of truth
  • Log aggregation across distributed services (replace Logstash)
  • Change Data Capture (CDC) from databases via Kafka Connect
Production examples: LinkedIn (born here), Uber (trip events), Netflix (activity tracking), Airbnb (event pipelines)
??

Choose RabbitMQ

Traditional Message Broker
  • Task / job queues where each message must be processed exactly once by one worker
  • Complex routing logic — route by content, headers, or multiple conditions
  • RPC (Remote Procedure Call) patterns with reply-to queues
  • You need sub-millisecond latency with push delivery to consumers
  • Delayed / scheduled message delivery (built-in plugin)
  • Priority queues — process urgent tasks before normal ones
  • Transient messages that don't need to be stored long-term
  • Teams wanting simple ops with a built-in management dashboard
Production examples: Instagram (notifs), Reddit (job queues), Mozilla (async tasks), VMware (event routing)
