01 Requirements Scoping

Always clarify functional vs non-functional requirements before designing. This signals structured thinking to the interviewer.

Functional Requirements

1
Upload video — users can upload short-form video (up to 60s/3min)
2
Feed / Recommendations — personalized infinite scroll video feed
3
Stream video — adaptive bitrate playback on mobile/web
4
Interact — like, comment, share, follow
5
Search — search by hashtag, sound, user, keyword
6
Notifications — real-time push for likes, comments, follows

Non-Functional Requirements

A
High Availability — 99.99% uptime (52 min downtime/year max)
B
Low Latency — video starts in <200ms, feed loads <100ms P99
C
Massive Scale — 1B DAU, 34M video uploads/day, 167M watch-hours
D
Eventual Consistency — likes/views can lag; feed freshness <5s
E
Durability — zero video loss (3x replication, geo-redundant)
F
Cost Efficiency — CDN tiering, cold storage for old content

2B Data Flow Walkthrough

Step-by-step trace of the two critical paths: Upload & Process, and Feed & Stream.

🔴 Upload & Process Path

1
Client chunks video into 5MB parts, requests pre-signed S3 URLs from Upload Service
2
Direct upload to Object Storage — bypasses app servers, reduces backend load
3
Upload Service confirms all chunks received, publishes video.uploaded event to Kafka
4
Transcoding Workers (Kafka consumers) pick up event, run FFmpeg to create HLS segments at 240p/480p/720p/1080p
5
Segments pushed to CDN origin, metadata (video_id, URLs, durations) saved to DB and cache
6
Notification sent to creator via push + Recommendation model is triggered to index new video

🔵 Feed & Stream Path

1
Client requests feed — authenticated request hits API Gateway → Feed Service
2
Feed Service checks Redis for pre-computed feed for this user (push model for <10K followers)
3
Cache miss → ML Recommendation Service pulls candidate videos, applies ranking model, returns top-N
4
Client receives feed (video IDs + CDN URLs), immediately starts prefetching next 3 videos
5
HLS manifest fetched from CDN edge — client picks right bitrate segment based on bandwidth
6
View event logged → Kafka → Analytics Service → updates recommendation signals for next feed

4B CAP Theorem Choices

Different subsystems make different CAP tradeoffs. Know these cold.

Subsystem CAP Choice Rationale Consistency Model
Video UploadCPMust not lose a video chunk; can tolerate upload failureStrong — idempotent chunk commits
Video Streaming / CDNAPAvailability + partition tolerance over consistency; old segment OKEventual — CDN TTL based invalidation
Like / View CountsAPApproximate counters acceptable; losing a like is tolerableEventual — Redis HLL + Cassandra counters
Feed ServiceAPSlightly stale feed OK; downtime is notEventual — pre-computed, TTL-based refresh
Auth / SessionsCPSecurity requires accurate session stateStrong — synchronous replication
User MetadataCPCorrect follower count, profile data mattersStrong — Postgres primary read
Search IndexAPNew videos appear in search within minutes, not secondsEventual — async Elasticsearch indexing
