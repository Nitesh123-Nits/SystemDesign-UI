Answer Framework

Amazon HLD interviews follow a structured arc. Never jump to solutions. This is the exact framework to ace it.

🎯 Amazon Interview Signal
Interviewers assess: Breadth (do you know the components), Depth (can you deep-dive), Trade-offs (do you justify choices), Scale awareness, and LP alignment (Customer Obsession, Think Big, Dive Deep).
1
Clarify Requirements (3–5 min)
"Before I start designing, let me clarify scope." Ask: What's the scale? Geo distribution? Are we building the catalog, checkout, fulfillment, or delivery tracking? Are we handling payments? What's the SLA? What's read vs write ratio? — This shows leadership, not just execution.
2
Back-of-Envelope Estimates (3–4 min)
Quantify the system before designing it. Derive QPS, storage, bandwidth. This grounds the design in reality and forces you to justify choices like "we need a CDN" or "we need sharding."
3
High-Level Design (8–10 min)
Draw the architecture: client → load balancer → API gateway → services → databases. Walk through the primary happy path (user places an order). Keep it high-level first, let the interviewer guide the depth.
4
Deep Dive on Critical Components (10–15 min)
Interviewers will direct: "Tell me more about inventory management" or "How does your search scale?" Have prepared deep-dives for: Order Service, Search, Inventory, Notification, Delivery Tracking.
5
Scalability & Bottlenecks (5 min)
Proactively address: single points of failure, hot partitions, N+1 queries, thundering herd, noisy neighbor, and how your design handles 10x traffic spikes (Prime Day, Black Friday).
6
Trade-offs & Alternatives (3 min)
Always compare your choice against the alternative: "I chose Kafka over SQS here because…" / "I'm using eventual consistency for the cart because strong consistency would add 40ms…" This is where senior candidates shine.

Core Services — Architecture

Each microservice owns its data, exposes APIs, and publishes/consumes events.

📦 Order Service — Design
  • Pattern: SAGA (choreography-based) — Order orchestrates inventory reservation, payment charge, fulfillment trigger via events
  • State machine: PENDING → CONFIRMED → PAID → FULFILLING → SHIPPED → DELIVERED → COMPLETED
  • Idempotency: Every order request carries a client-generated idempotency key. Prevents duplicate orders on retry
  • Database: PostgreSQL, sharded by user_id hash. Each shard has 1 primary + 2 read replicas
  • Outbox pattern: Write order to DB and event to outbox table atomically. Separate relay process publishes to Kafka
⚠️ SAGA compensation: If payment fails after inventory reserved, publish inventory.release event. If fulfillment fails, publish payment.refund event. Design for failure.
// Order state machine transitions PENDING → inventory.reserved → CONFIRMED → payment.captured → PAID → fulfillment.accepted → FULFILLING → carrier.picked_up → SHIPPED → delivery.confirmed → DELIVERED // Compensation (rollback) flows payment.failed → inventory.release(orderId) → order.status = CANCELLED fulfillment.rejected → payment.refund(paymentId) → inventory.release(orderId) → order.status = CANCELLED // Outbox pattern (atomicity) BEGIN TRANSACTION INSERT INTO orders (...) INSERT INTO outbox (event_type, payload) COMMIT -- single atomic write
📊 Inventory Service — Concurrency
  • Challenge: Flash sale — 50K concurrent users try to buy last 1,000 units. Race conditions cause overselling
  • Solution 1 — Optimistic Locking: Read inventory with version number. Update only if version matches. Retry on conflict. Works well for low-contention items
  • Solution 2 — Redis Atomic DECR: Pre-load hot inventory into Redis. Use DECR + Lua script atomically. Sync to DB asynchronously. Works for flash sale spikes
  • Solution 3 — Token Bucket: For ultra-hot items, pre-issue purchase tokens. Only token holders can buy. Reduces DB load entirely
-- Optimistic locking UPDATE inventory SET quantity = quantity - 1, version = version + 1 WHERE sku_id = 'SKU123' AND quantity > 0 AND version = 42; -- If 0 rows affected: conflict, retry -- Redis atomic reservation (Lua) local qty = redis.call('GET', KEYS[1]) if tonumber(qty) >= tonumber(ARGV[1]) then redis.call('DECRBY', KEYS[1], ARGV[1]) return 1 -- success end return 0 -- insufficient stock
🔍 Search Service Architecture
  • Stack: Elasticsearch 8.x with custom analyzers. BM25 relevance scoring
  • Index design: 6 primary shards × 2 replicas. Each shard ~2TB. Sharded by hash routing
  • Write path: Product updates → Kafka topic → Flink stream job → incremental ES index updates. Near-real-time (~5s delay)
  • Query path: API → Search Service → ES. Results merged with personalization scores (ML model in Redis)
  • Facets: Pre-aggregated facet counts cached. Avoids expensive aggregation queries at read time
  • Spelling correction: Did-you-mean using fuzzy matching + query log mining

End-to-End Order Flow

Walk through every hop from "Place Order" click to "Order Confirmed" — the critical happy path interviewers ask about.

T+0ms
User clicks "Place Order"
Mobile/Web app sends POST /orders with Idempotency-Key header. Client shows optimistic loading state.
T+5ms
WAF + Load Balancer + API Gateway
WAF validates request (rate limit, IP reputation). LB routes to least-loaded API Gateway node. Gateway validates JWT, extracts userId, routes to Order Service. Circuit breaker checked — if Order Service unhealthy, returns 503 immediately.
T+15ms
Order Service — Validation
Checks idempotency key in Redis (SETNX). If duplicate: return cached response. Validates cart items exist. Calls Pricing Service to confirm prices (prevents cart price tampering).
T+30ms
Inventory Reservation
Calls Inventory Service synchronously (internal gRPC). Redis Lua script atomically decrements inventory. If insufficient stock → 409 Conflict to user. On success, inventory status = RESERVED (not deducted yet).
T+60ms
Payment Authorization
Calls Payment Service. Payment Service calls Stripe/Braintree with authorize-only (not capture). Stores payment token. If payment fails, publish inventory.release event → compensating transaction.
T+80ms
Order Persisted + Event Published
Order written to DB + outbox table in a single ACID transaction. Relay process reads outbox and publishes order.placed to Kafka topic. Payment captured asynchronously.
T+85ms
Response to Client
Order Service returns HTTP 201 with orderId and status=CONFIRMED. Client shows "Order placed!" state. Everything after this is async.
T+500ms (async)
Downstream Events (Async)
Kafka consumers: (1) Notification Service → push + email to user. (2) Fulfillment Service → assigns warehouse, picks, packs. (3) Analytics → updates dashboards. (4) Recommendations → re-trains user profile. None of these block the user.
✓ Why this design scales: The synchronous critical path is only: Gateway → Validation → Inventory → Payment → DB write → respond. Everything else (notifications, fulfillment, analytics) is decoupled via Kafka. The user gets <100ms response even if notification service is down.

Search & Product Catalog at Scale

Search is the most critical path for revenue. 40% of purchases start from search.

Write Path
Product Update → Search Index Sync
Seller updates product → Product DB → CDC (Debezium) captures WAL changes → Kafka product.updated event → Elasticsearch indexer consumer → Bulk upsert to ES index. Latency: ~5s eventual consistency. Acceptable for catalog; not needed for inventory.
Read Path
Search Query Execution
Client query → API Gateway → Search Service → Redis check (popular queries cached 60s) → If miss: ES multi_match query with filters + personalization boost → Merge with ML ranking signal → Return paginated results (cursor-based, not offset).
Scale Problem
500M Products, 35K QPS Search
Single ES node can handle ~5K QPS. Need 6 primary shards × 2 replicas = 18 shards across 15+ nodes. Route reads to replicas only. All writes go to primary, auto-replicated. Use routing parameter to collocate related docs on same shard (by category).
Typeahead / Autocomplete
  • Trie in Redis: Top 1M queries stored as Redis sorted sets with prefix keys. O(1) lookup for prefix suggestions
  • Debounce: Client waits 200ms after keystroke before firing autocomplete API. Reduces QPS by 5×
  • Personalized: Top-k suggestions re-ranked by user's past categories using user embedding similarity
  • ES Completion suggester: Fallback for long-tail queries not in Redis trie
Why Not SQL Full-Text Search?
⚡ PostgreSQL full-text search degrades at 500M rows. No native faceting, no fuzzy matching, no ranking. Elasticsearch is purpose-built: inverted indices, BM25, aggregations, sharding — all built-in at petabyte scale.

Why This Platform Scales

Six architectural pillars that make an e-commerce platform handle Amazon-level scale. These are the "why" questions interviewers probe.

1. Stateless Services + Horizontal Scaling
All API services are stateless (no sticky sessions). User state lives in Redis/DB, not in-process. Any instance can serve any request. Auto Scaling Groups add/remove instances in seconds based on CPU/RPS. 10× capacity in ~3 minutes.
2. Read-Heavy Caching
100:1 read/write ratio means caching is king. Multi-layer: CDN (static assets, 90%+ hit), Redis L1 (hot products/sessions, 95% hit), DB read replicas (remaining reads). Only writes and cache misses hit primary DB.
3. Database Sharding
Orders sharded by user_id (consistent hash). Products sharded by category hash. Each shard is independently scalable. No cross-shard joins in hot paths. Shard count pre-planned to allow splits without resharding (consistent hashing with virtual nodes).
4. Async Event-Driven Architecture
Kafka decouples producers from consumers. Order Service doesn't wait for Notification, Fulfillment, or Analytics. Downstream services can be slow, fail, or be upgraded without affecting the checkout path. Back-pressure naturally handled by consumer lag.
5. Microservices Domain Isolation
Each service owns its DB — no shared state means each service scales independently. A spike in search doesn't affect checkout. Catalog Service can be read-scaled to 100 instances without touching Order Service. Failure isolation prevents cascading.
6. CDN + Edge Offloading
Product images (2PB!) and static assets served from CDN. 90%+ traffic never reaches origin. GeoDNS routes users to nearest edge PoP. P99 latency drops from 200ms to <20ms for cached content globally. Origin servers only handle dynamic API calls.

Reliability & Fault Tolerance

How to achieve 99.99% uptime — the patterns that prevent and contain failures.

Pattern 1
Circuit Breaker (Resilience4j / Hystrix)
If Payment Service fails 5 consecutive times in 10s, circuit opens. All subsequent calls fast-fail for 30s. Prevents cascading — a slow payment gateway doesn't bring down checkout. Auto-recovers (half-open probe after timeout).
Pattern 2
Bulkhead Isolation
Separate thread pools for different downstream calls. If Recommendation Service is overwhelmed, it can't exhaust the thread pool used by Order Service. Each service gets a fixed resource budget.
Pattern 3
Retry with Exponential Backoff + Jitter
Transient failures (network blip, DB hiccup) auto-retried. Base delay 100ms × 2^attempt + random jitter (0–100ms) prevents thundering herd on retry storms. Max 3 retries; non-idempotent calls (payment) only retried with idempotency key.
Pattern 4
Multi-Region Active-Active
Deployed in 3 AWS regions (us-east-1, eu-west-1, ap-south-1). Route53 latency routing. If us-east-1 has an outage, GeoDNS shifts traffic to eu-west-1 within 60s. Data replication: Order DB uses synchronous replication within region, async cross-region.
Graceful Degradation — What Degrades First
DEGRADE 1
Recommendations offline → Show popular items
User still browses and buys. Revenue impact minimal.
DEGRADE 2
Reviews/Ratings unavailable → Hide section
Product pages still load fully without review panel.
DEGRADE 3
Real-time tracking down → Show last known ETA
Stale ETA better than error. User experience preserved.
NEVER DEGRADE
Checkout, Payments, Order confirmation
These are revenue-critical. Circuit break anything around them before touching core order flow.

99.99% uptime = 0.01% downtime
Per year: 0.0001 × 365 × 24 × 60 = 52.6 min downtime budget
Per month: ~4.4 minutes
📐 To hit 99.99%, you need: redundant everything (N+1 min), automated failover <30s, chaos engineering, RTO <1min, RPO=0 for orders.

Caching is the single biggest lever for read-heavy platforms. Get this right and you need 10× fewer DB servers.

Write-Through Cache
On product update: write to DB AND update Redis simultaneously. Always consistent. Slight write latency increase. Used for: product prices (must be accurate), inventory counts.
Consistent Slower writes
Cache-Aside (Lazy)
Read hits cache → return. Miss → read from DB → populate cache → return. Write → DB only, invalidate cache key. Used for: product catalog, reviews (eventual consistency OK).
Flexible First miss cold
Event-Driven Invalidation
Product update → Kafka event → Cache invalidation consumer deletes/updates Redis keys across all regions. Ensures global cache consistency after a product update propagates in <2s.
Global Eventually consistent
⚠️ Cache Stampede / Thundering Herd
When a popular product's cache entry expires, 10,000 simultaneous requests miss and all hit the DB. Solution: Use a mutex lock (Redis SETNX) — first request gets the lock, fetches from DB, populates cache. Others wait and serve the populated cache. Alternatively, use "probabilistic early expiration" — randomly re-compute cache slightly before TTL expires in background.
// prime day / black friday

Handling Traffic Spikes

Prime Day 2023: 375 million items sold. 10× normal traffic in minutes. How do you prepare?

Pre-Event Preparation (Days Before)
Pre-Scaling & Pre-Warming
1. Scale up minimum instances to 3× normal (don't rely on autoscale lag). 2. Pre-warm Redis caches with anticipated hot products. 3. Pre-provision Kafka partitions (easier to add before peak than during). 4. Run load tests at 5× projected peak. 5. Freeze deployments 48h before event.
During Event
Load Shedding + Queuing
1. API Gateway enforces per-user rate limits (100 req/min for browse, 10 req/min for checkout). 2. Virtual waiting room queue for checkout during extreme spikes — users get a position number, enter checkout sequentially. 3. Read-only mode: if inventory service is overwhelmed, disable stock level display (show "In Stock" vs exact count). 4. Drop analytics events before dropping orders.
Flash Sale Architecture
Token-Based Purchase Control
For limited-quantity items (1,000 units, 1M trying to buy): 1. Pre-generate 1,000 purchase tokens. 2. On "Buy Now" click, atomically claim a token from Redis (SPOP). 3. Only token holders proceed to checkout. 4. Others see "Sold Out" immediately. 5. Eliminates inventory contention entirely — 999,000 users don't even reach the DB.

Users see "Processing your order..." screen. Orders processed in FIFO order. Decouples user experience latency from backend throughput — the queue absorbs the spike.

Counter Questions & Answers

The follow-up questions Amazon interviewers always ask. Be ready for all of these.

Q How do you prevent overselling when 10,000 users try to buy the last unit simultaneously? +
Three approaches, I'd use based on contention level:

1. Optimistic locking (low-medium contention): Read inventory with version number. UPDATE WHERE version=N. If 0 rows affected, retry. Works when concurrent buyers are in hundreds.

2. Redis atomic DECR with Lua (high contention): Pre-load hot inventory counts in Redis. Atomic DECRBY via Lua script (check + decrement is atomic). Sync to DB asynchronously. Handles 50K+ concurrent buyers on a single item — Redis does 1M ops/sec.

3. Pre-issued tokens (flash sales): For ultra-hot items like limited-edition drops, pre-generate N tokens. Buyers atomically claim tokens via Redis SPOP (atomic set pop). Zero inventory DB writes until token claimed. Eliminates race condition entirely.

Trade-off to mention: Redis approach risks inconsistency if Redis fails before DB sync — use Redis persistence (AOF) + periodic reconciliation job.
Q How do you ensure an order is not placed twice (duplicate prevention)? +
Idempotency key pattern. Client generates a UUID before sending POST /orders and includes it as Idempotency-Key header. Order Service does:

1. Redis SETNX("idem:{key}", "processing", TTL=24h) — if returns 0, key already exists
2. If exists: return cached response (don't process again)
3. If new: process order, store result in Redis with same key
4. Any retry (network timeout, client crash) hits the cached response

Also store idempotency key as UNIQUE constraint in Orders DB as second line of defense — handles Redis failure.
Q How does search stay consistent when sellers update products frequently? +
CDC + Kafka + ES incremental update pipeline. I accept eventual consistency (~5s delay) for catalog data — it's not critical path.

1. Debezium (CDC tool) reads PostgreSQL WAL → publishes product.updated events to Kafka
2. ES Indexer Service consumes from Kafka, batches 1,000 doc updates, bulk-upserts to Elasticsearch
3. For price changes specifically: synchronous write-through to both DB and ES (more important to be fast)

Why not sync ES write on every product save? ES bulk indexing throughput is high, but if we synchronously block seller's save on ES write success, a slow ES cluster degrades seller experience. Async is better here.
Q What happens if the payment service is down during checkout? +
Circuit breaker + fallback strategy.

1. API Gateway has circuit breaker on Payment Service endpoint (Resilience4j). After 5 failures, circuit opens for 30s.
2. During open circuit: show user "Payment processing is temporarily slow, your order is being held." — enqueue payment retry in Kafka.
3. Order goes to PAYMENT_PENDING state, inventory stays reserved (with TTL = 30 min).
4. Separate payment retry worker processes pending payments when circuit closes.
5. If payment fails after 3 retries: cancel order, release inventory, notify user.

Key insight: Don't return 500 to user. Show a graceful message and hold the order. This preserves UX and Amazon LP "Customer Obsession."
Q How do you handle multi-item orders where one item is out of stock after the order is placed? +
SAGA with partial fulfillment option.

Before order submission: inventory reservation is atomic for the batch. If any item fails to reserve, the entire reservation is rolled back — user shown "Item X is out of stock." This is the simplest path.

For post-payment discovery (rare, item physically missing at warehouse):
1. Fulfillment service publishes item.unavailable event
2. Order Service transitions to PARTIAL_FULFILLMENT state
3. Business rule: if item is <10% of order value → ship rest, partial refund automatically
4. If >10% → contact user with options: wait for restock, substitute, or refund

This maps to Amazon's actual "We're sorry, one item in your order is delayed" experience.
Q How would you handle a database hotspot on a popular product? +
Hot partition problem — three solutions:

1. Read replicas + aggressive caching: Route all reads for popular products to Redis cache (TTL=5min). Only cache miss hits DB. A product with 100K views/min → 5K hits/min to DB (cache miss rate 5%). Acceptable.

2. Write serialization for inventory: Hot product inventory updates go through a single-thread Redis Lua queue. Redis serializes the operations. DB gets batched writes every 100ms instead of 100K concurrent writes.

3. Read-your-writes relaxation: For cart and inventory display, accept eventual consistency. User sees "~500 remaining" instead of exact count. Dramatically reduces write pressure.

Shard splitting: If a product category becomes a consistent hotspot (e.g., "Electronics"), split that shard. Plan for this upfront with consistent hashing so splits are possible without full resharding.
Q How do you scale the cart service for 100M DAU? +
Redis Hash with user_id as key.

Cart data structure: Redis Hash — key = "cart:{userId}", field = skuId, value = JSON{qty, price, addedAt}

Operations: HSET (add item), HGET (get item), HDEL (remove), HGETALL (view cart) — all O(1) or O(N items)

Scale: 100M active carts × 500 bytes avg = 50GB. Fits in Redis cluster. Cart reads are user-scoped (no cross-user aggregation), so hash-slot sharding by user_id distributes perfectly.

Persistence: Redis AOF enabled. If Redis fails, carts are recovered. Additionally, sync carts to PostgreSQL every 5 min for durability — users never lose cart even if Redis cluster fails.

Guest carts: Session ID as key. TTL=7 days. On login, merge guest cart with user cart.
Q How would you design real-time order tracking that scales to 1M concurrent deliveries? +
WebSocket gateway + Redis Pub/Sub + Cassandra.

Driver app sends location every 5s via WebSocket. WS Gateway publishes to Redis Pub/Sub channel "location:{orderId}". Customer's browser subscribes to same channel via WS or SSE.

Scale math: 1M active deliveries × 1 update/5s = 200K events/sec. Redis Pub/Sub handles 1M+ msg/sec on a single node. WS Gateway is horizontally scalable (stateless with shared Redis channel awareness).

Storage: Cassandra stores time-series location data. Partition key = (driver_id, date). Clustering key = timestamp. Supports efficient time-range queries for trip playback.

Long polling fallback: For customers on weak mobile connections, provide /orders/{id}/location REST endpoint (long-poll, 30s timeout). Less efficient but more compatible.
Q Why microservices over a monolith for this system? +
Honest trade-off answer — don't just say microservices are better.

In favor of microservices here:
1. Independent scaling: Search Service needs 30 ES nodes; Order Service needs 10 app servers. With a monolith, you scale everything together — wasteful.
2. Independent deployment: 500-engineer org. Team A deploys catalog updates without coordinating with Team B (payments). Conway's Law in action.
3. Technology fit: Cart → Redis, Search → ES, Orders → PostgreSQL, Location → Cassandra. Each data store is best-of-breed for its access pattern.
4. Failure isolation: Payment gateway outage doesn't crash the browse experience.

Costs to acknowledge: Distributed system complexity (distributed transactions, eventual consistency), network latency overhead between services, operational complexity (service mesh, observability). "I'd start with a modular monolith in early-stage, extract to microservices when team/scale demands it."
Q How do you handle the CAP theorem for this system? Where do you choose consistency vs availability? +
Different components choose differently — context-driven.

CP (Consistency over Availability):
• Payment processing — must not double-charge. Use 2-phase commit within payment service. Accept brief unavailability over incorrect charge.
• Inventory reservation — must not oversell. Synchronous DB write with locking. Accept higher latency.

AP (Availability over Consistency):
• Product catalog — eventual consistency fine. Stale price for 5s is acceptable. Use Redis cache + eventual DB sync.
• Search index — 5–10s delay OK. Seller cares about eventual indexing, not 0ms.
• Cart contents — eventual is fine. Cart synced to DB asynchronously.
• Review counts — "1,234 reviews" vs "1,235 reviews" doesn't matter. Eventual.

Key phrase to say: "I apply strong consistency selectively — only where incorrect data has a direct financial or correctness consequence. Everything else gets eventual consistency for availability."
Q How would you improve this design if you had 6 months? +
This is a "Think Big" Amazon LP question. Show you can envision the future state.

1. ML-powered dynamic pricing: Real-time price optimization based on demand signals. Requires event stream from browsing, competitor data, inventory level — Flink streaming job feeding a pricing model.

2. Predictive pre-scaling: Instead of reactive autoscaling, predict traffic spikes from historical patterns + external signals (marketing campaigns, weather). Pre-scale 30 min before spike, not during.

3. GraphQL API Gateway: Replace REST with GraphQL for frontend. Eliminates N+1 fetch problem on product pages (single query fetches product + inventory + reviews + recommendations). Reduces mobile data usage 40%.

4. Event sourcing for Orders: Instead of mutable order state, store immutable event log. Full audit trail. Easy replayability. Enables time-travel debugging.

5. CQRS for high-read services: Separate read and write models. Write model normalized for integrity; read model denormalized for speed (pre-joined, pre-aggregated).
